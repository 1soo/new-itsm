# CLAUDE.md

서비스 요청(srm) 도메인 애플리케이션 서비스 계층.

## 파일
- `ServiceRequestService.java` — 요청 생성·상태전이·배정·댓글·CSAT·지식연계·자산연계 조회(REQ-ITAM-006, TicketLink 재사용) 유스케이스. 공용 승인 게이트(`common.approval.application.ApprovalGateService`) 연동(2026-07-11 승인 프로세스 커스텀 기능, 승인 결정·대기 목록은 이 서비스가 아닌 공용 엔진이 담당). 2026-07-22 상태별 승인자 지정 확장: `transition()`은 target 가드 없이 무조건 `checkGate(domain=SERVICE_REQUEST, requestSubtypeKey=String.valueOf(catalogItemId), requesterId=principal.userId(), TT, id, targetState=target)` 호출(SUBMITTED→VALIDATED→ROUTED→IN_FULFILLMENT→FULFILLED→CLOSED 어떤 전이든 게이트 대상, checkGate의 requesterId를 호출자로 통일 — 기존 `sr.getRequesterId()`(티켓 고정 필드) 참조가 잘못된 산출 방식이었음이 이번 유지보수로 확인·수정됨). `create()`는 `TicketCreationGateSupport.createThenGate`로 REQUIRES_NEW 분리(생성시점 targetState="SUBMITTED"). `assigneeCandidates()`(API-SRM-017, 2026-07-15): 카탈로그 항목의 assigneeRoleId 보유+ACTIVE 사용자 목록(`AppUserRepository.search(roleCode 필터)` 재사용, 역할 미지정 시 빈 배열). `transition()`: target=ROUTED이고 담당자 미배정이면 `ASSIGNEE_REQUIRED_FOR_ROUTING`(409, vulnerability의 ASSIGNEE_REQUIRED_FOR_REMEDIATION과 동일 패턴), 승인 게이트 체크보다 먼저 판정. `assertCanView`: 기존 정적 APPROVER 전체조회 폐지, AGENT/PROCESS_OWNER 또는 `ApprovalGateService.canApproverView(SERVICE_REQUEST, catalogItemId, requesterId)` 매칭 시에만 조회 허용(2026-07-15 승인 대상자 동적 상세조회 권한, 이번 유지보수로 변경 없음 — canApproverView는 티켓의 실제 요청자 기준 유지). 상세 조회의 `approval`(`ApprovalInfo`)에 `targetState` 포함, 목록은 `pendingApprovalTargetStatesOf` 배치 조회로 `RequestSummaryResponse.pendingApprovalTargetState` 채움(N+1 방지). `create()`: 제출 시 카탈로그 항목의 `formSchema`(자체 8×n 그리드) + `formValues`를 공용 `common.form.FormSubmissionValidator`(2026-07-18 유지보수 요청, form.io 완전 제거)로 재검증 후 `ServiceRequest.formValues`(JSONB)에 그대로 저장. `detail()`: 저장된 `formValues` JSONB를 그대로 파싱해 반환(기존 EAV 조립 로직 제거). `list()`: `queueId` 대신 `categoryId`/`uncategorized`로 필터(2026-07-18 유지보수 요청 — 요청 큐 폐지, `service_catalog_item.category_id` 실시간 조인). 신규 `categoryCounts()`(API-SRM-016): `ServiceCatalogCategoryRepository`를 sortOrder 오름차순 순회하며 카테고리별 미종료 건수 집계 후 미분류 그룹을 마지막에 추가(기존 큐 목록·건수 조회 `QueueService` 완전 대체)
- `ServiceCatalogService.java` — 서비스 카탈로그 항목 CRUD·동적 양식 관리(승인 필요 여부 필드는 제거됨). 담당자 역할(assigneeRoleId, 2026-07-15) CRUD 지원, 상세 응답에 `RoleRepository`로 역할명(assigneeRoleName) resolve해 포함. `category`(자유 텍스트) → `categoryId`(FK) 전환(2026-07-16 유지보수 요청) — 생성/수정 시 `ServiceCatalogCategoryRepository`로 존재 검증(404 CATEGORY_NOT_FOUND), 목록/상세 응답에 `categoryName` resolve해 포함. `formSchema`(자체 8×n 그리드 스키마, `{"components":[]}` 기본값)를 `ServiceCatalogItem.formSchema`(JSONB)에 그대로 저장·조회(2026-07-18 유지보수 요청, form.io 완전 제거). `queueId` 처리 로직은 2026-07-18 유지보수 요청(요청 큐 폐지)으로 제거됨
- `ServiceCatalogCategoryService.java` — 서비스 카탈로그 카테고리 CRUD(2026-07-16 유지보수 요청). list — sortOrder 오름차순 + 카탈로그 항목 참조 수(itemCount), create/update — 이름 중복 409(CATEGORY_NAME_DUPLICATE), delete — `ServiceCatalogItemRepository.countByCategoryId`로 참조 중이면 409(CATEGORY_IN_USE, 자동 미분류 처리 안 함)
- `SrmApprovalTicketSummaryProvider.java` — 공용 승인 대기함·상세(API-COM-003/004)가 노출할 SERVICE_REQUEST 티켓 요약(ticketKey·카탈로그명·요청자명) 어댑터(`ApprovalTicketSummaryProvider` 구현)
- `SrmApprovalRequestSubtypeProvider.java` — 승인 프로세스 관리자 CRUD(API-AUTH-024)의 SERVICE_REQUEST 요청유형 후보(서비스 카탈로그 항목) 어댑터(`ApprovalRequestSubtypeProvider` 구현)
- `MetricsService.java` — SRM 지표(CSAT·SLA 등) 집계
- `RequestStateMachine.java` — 수동 상태 전이 규칙(package-private, SUBMITTED→VALIDATED→ROUTED→IN_FULFILLMENT→FULFILLED→CLOSED)
- `SlaCalculator.java` — 기한 대비 잔여 시간으로 SLA 상태(OK/WARNING/BREACHED) 산정(package-private)
