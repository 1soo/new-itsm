# 용어집 (Glossary)

도메인 용어(엔티티명·상태값·약어 등)를 정의한다. `analyzer`가 요구사항 분석 중 신규 용어를 정의하고, `designer`가 설계 중 구체화·추가한다. 간결하게 유지하며 정의가 바뀌면 그 자리에서 갱신한다(이력 남기지 않음).

- **승인요청자 역할(Requester Role Scope)**: `approval_process_requester_role`. 승인 프로세스 규칙이 매칭되는 대상을 좁히는 스코프 축 — 티켓의 **요청자**가 보유한 역할이 이 스코프에 하나라도 포함되어야 규칙이 매칭된다(ANY, 0개면 요청자 무관). (승인 프로세스 공용 엔진)
- **승인자 역할 / 승인 대상자(Approver Step Role)**: `approval_process_step_role`. 승인 프로세스 규칙의 각 차수(step)에 지정된, **실제로 승인/반려를 수행할** 역할 — "승인 대상자로 할당된 역할"은 이 축을 가리킨다. 승인요청자 역할과는 별개 축(전자는 "누구의 요청에 적용되는가", 후자는 "누가 승인하는가"). (승인 프로세스 공용 엔진)
- **동적 상세조회 권한(Dynamic Detail-View Authorization)**: 승인자 역할이 어떤 티켓의 상세를 조회할 수 있는지를, 고정 역할 목록이 아니라 해당 티켓의 (도메인, 요청유형, 요청자 보유 역할)에 매칭되는 `approval_process` 규칙의 승인자 역할(step role) 보유 여부로 동적으로 판정하는 방식. 실제 승인 인스턴스(`approval_request`) 생성 여부와 무관하게 규칙 설정 자체와의 매칭만으로 판정한다(2026-07-15 유지보수 요청, SRM/CHANGE의 기존 정적 "APPROVER=도메인 전체조회" 권한을 대체하고 INCIDENT/PROBLEM/ASSET/VULNERABILITY/COMPLIANCE/ESM까지 확장). (승인 프로세스 공용 엔진, `common/approval` 캡슐화)
- **카탈로그 담당자 역할(Catalog Assignee Role)**: `service_catalog_item.assignee_role_id`. 요청 유형(카탈로그 항목)에 지정하는 처리 담당 역할(선택). 자동 배정에 쓰이지 않고, 상담원이 라우팅/배정 시점에 이 역할 보유자 후보 목록 중에서 수동으로 담당자를 선택하는 데에만 사용된다. (서비스 요청 관리)
- **미분류(Unclassified)**: 서비스 요청의 카탈로그 항목에 카테고리(`service_catalog_item.category_id`)가 지정되지 않은 상태. 요청 처리함 화면(SCR-SRM-004)에서 "미분류" 그룹으로 표시된다(2026-07-18 유지보수 요청, 처리 큐 축 폐지 — 기존 `queue`/`queue_id` 기반 정의를 카테고리 기반으로 대체). (서비스 요청 관리)
- **카탈로그 카테고리(Catalog Category)**: `service_catalog_category`. 서비스 카탈로그 항목을 분류하는 고정 목록(관리자(PROCESS_OWNER)가 SCR-SRM-009에서 추가/수정/삭제). 카탈로그 항목은 `service_catalog_item.category_id`로 이 목록의 항목 하나를 참조한다. (서비스 요청 관리)
- **전이 버튼 라벨(Transition Button Label)**: 티켓 상세 화면의 상태 전이 버튼에 표시하는 문구. 도착 상태명(예: "이행중")이 아니라 그 상태로 가기 위한 동작 동사형(예: "이행 시작")을 사용한다. 전이 완료 토스트 문구는 도착 상태명(`statusLabel`)을 사용해 결과를 안내하므로, 버튼 라벨(동작 예고, `transitionLabel`)과는 별개 함수로 관리한다. (공통, SRM/INCIDENT/PROBLEM/CHANGE/VULNERABILITY/ASSET/ESM 7개 도메인)
- **동적 양식 스키마(Form Schema)**: `service_catalog_item.form_schema`(JSONB). SRM 전용 자체 8×n 그리드 스키마(`components` 배열 + 최상위 `labels` 배열 — 각 컴포넌트에 type/key/그리드 위치·크기/Content 설정을 포함, 중첩 레이아웃 없음)를 저장한다. 팔레트는 9종(text/textarea/select/radio/checkbox/date/file/guide-text/guide-file)이며, 7개 입력 타입만 값 입력·`input.placeholder`·정렬(9방향)을 갖고(`radio`/`checkbox`는 추가로 옵션 배치 방향·여백을 가짐, `text`만 정규식 검증 대상) `guide-text`(정적 텍스트)·`guide-file`(정적 첨부 가이드 파일, base64 인라인)는 값 입력이 없어 제출 데이터 대상에서 제외된다. 그리드에 직접 배치하던 `label` 타입은 폐기되고 [그리드 라벨(태그)](#그리드-라벨태그-grid-label-tag)로 대체됐다(2026-07-18 유지보수 요청 4차). 관리자는 "Form 설정" 팝업(팔레트/캔버스 2분할, 캔버스=미리보기 통합, [screen/service-request.md](../02_plan/screen/service-request.md) 5절)으로 편집한다. ESM은 이 스키마를 사용한 적이 없으며 기존 행 기반 EAV(`esm_catalog_form_field`)를 그대로 사용한다. (서비스 요청 관리 전용)
- **그리드 라벨(태그, Grid Label/Tag)**: `service_catalog_item.form_schema.labels`(JSONB 배열, 각 항목 id/text/textColor/borderColor). 과거 그리드에 직접 배치하던 정적 `label` 컴포넌트를 대체하는 개념으로, 컴포넌트가 아니라 **다른 컴포넌트(7개 입력 타입 + `guide-text`/`guide-file`)에 선택적으로 부여하는 태그**다(컴포넌트의 `labelId`로 참조, 2026-07-18 유지보수 요청 4차). 같은 라벨을 2개 이상의 컴포넌트가 참조하면 빌더 캔버스가 그 컴포넌트들 위치의 합집합 경계에 `borderColor` 테두리를 오버레이로 그린다(1개 이하 참조 시 테두리 없음). (서비스 요청 관리 전용, [screen/service-request.md](../02_plan/screen/service-request.md) 5.8절)
- **폼 제출 데이터(Form Values / Submission Data)**: `service_request.form_values`(JSONB). 제출된 값을 컴포넌트 `key` 기준 key-value 맵으로 그대로 저장한다(레이아웃 엔진과 무관, 필드별 집계 로직 없음). ESM은 별도로 행 기반 EAV(`esm_request_form_value`)를 사용하며 이 컬럼과 무관하다. (서비스 요청 관리 전용)
