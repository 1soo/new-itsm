# 문제 관리 (Problem Management)

> **인시던트의 근본 원인(root cause)을 식별·제거하여 반복 발생을 예방하는 ITSM 핵심 실천법.** "무엇이 장애를 일으켰는가"를 넘어 "그 원인을 만든 조건과 기여 요인은 무엇인가"를 묻는 것이 문제 관리의 본질이다.

---

## 📌 개요 (무엇을 & 왜)

**문제 관리(Problem Management)**는 IT 서비스에서 발생하는 인시던트의 원인(또는 잠재적 원인)을 식별하고 관리하는 프로세스로, ITIL을 비롯한 ITSM 프레임워크의 핵심 구성 요소다.

- **무엇인가(What):** 인시던트를 분석하고 근본 원인을 찾아 장기적 해결책 또는 임시 대응책(workaround)을 구현하는 실천법.
- **왜 중요한가(Why):** 서비스 데스크의 소음(noise)을 줄이고, 서비스 신뢰성을 높이며, 팀을 끊임없는 "불끄기(firefighting)"에서 해방시킨다.

> **핵심 통찰:** 인시던트는 서비스가 복구되면 "종료"되지만, 근본 원인과 기여 요인이 해결되기 전까지 **문제(problem)는 여전히 남아 있다.** 설정 파일이 재작성되거나 DB 항목이 손상된 것이 "원인"이라 해도, 진짜 전문가는 "어떻게 그 설정 파일이 재작성될 수 있었는가?", "어떤 조건이 DB 손상을 만드는가?"를 묻는다.

문제 관리는 단절된 사일로(silo) 팀이 수행하는 별개의 작업이 되어서는 안 된다. IT·보안·소프트웨어 개발 등 **여러 팀에 걸쳐 상시적으로, 폭넓게** 실천되어야 가치를 발휘한다.

### 기대 효과 (Benefits)

| 효과 | 설명 |
|------|------|
| ⏱️ **해결 시간 단축** | 오늘의 인시던트 이면의 문제를 파악하면 미래 인시던트에 더 빠르게 대응. 문제 분석 모범 사례를 코드화(codify)하여 다음 장애 시 신속 대응. |
| 💰 **값비싼 인시던트 회피** | Gartner에 따르면 많은 조직이 다운타임 비용을 **시간당 30만 달러($300,000) 이상**으로 보고. 웹 기반 서비스는 훨씬 더 높을 수 있음. |
| 📈 **생산성 향상** | 인시던트 대응 빈도를 줄여, 새로운 고객 가치를 만드는 데 시간과 자원을 되돌림. |
| 🧠 **근본 원인 학습 역량 강화** | 팀이 지속적으로 조사·학습하고 가치 있는 업데이트를 배포. (단, 운영과 동떨어진 사일로형 문제 관리팀은 이 효과를 저해함) |
| 🔄 **지속적 서비스 개선(CSI) 촉진** | 저성능을 유발하던 인시던트를 고치면서 동시에 서비스 품질 개선을 함께 배포. |
| 😀 **고객 만족도 증가** | 반복 인시던트가 줄면 고객 신뢰가 쌓임. 같은 장애가 반복되면 고객 인내심은 빠르게 소진됨. |

## 🧩 핵심 개념 및 용어

| 용어 | 정의 |
|------|------|
| **인시던트(Incident)** | ITIL 정의: "IT 서비스의 계획되지 않은 중단, 또는 IT 서비스 품질의 저하." 서비스가 다시 작동하면 종료됨. |
| **문제(Problem)** | ITIL 정의: "하나 이상의 인시던트의 원인, 또는 잠재적 원인." 근본 원인이 해결될 때까지 존속. |
| **근본 원인(Root Cause / RCA)** | 문제와 관련된 인시던트들의 최초 원인. **단, 근본 원인은 보통 하나가 아니다** — 여러 기여 요인(contributing factors)을 총체적으로 고려해야 함. |
| **알려진 오류(Known Error)** | ITIL 정의: "문서화된 근본 원인과 임시 대응책(workaround)을 가진 문제." 이를 기록해 두면 향후 인시던트 발생 시 다운타임이 줄어듦. |
| **알려진 오류 데이터베이스(KEDB)** | 알려진 오류(근본 원인 + 워크어라운드) 정보를 저장하는 문서/저장소. |
| **워크어라운드(Workaround)** | 문제의 영향을 줄이고 인시던트로 번지는 것을 막는 **임시 해결책.** 이상적이진 않지만 비즈니스 영향과 고객 대면 장애를 제한. |
| **서비스 요청(Service Request)** | ITIL 정의: "사용자가 무언가를 제공받기 위한 공식 요청" (정보/조언, 비밀번호 재설정, 신규 사용자 워크스테이션 설치 등). ITIL V3(2007) 이전에는 인시던트와 구분되지 않고 모두 "인시던트"로 묶였음. |

### 🔑 문제 vs 인시던트

- 최근 배포를 **롤백(rollback)**하면 서비스는 복구되고 인시던트는 끝나지만, **근본 원인(문제)은 그대로 남는다.**
- 인시던트 관리와 문제 관리의 행동 양식은 유사하고 겹치는 부분이 많지만, 지향점이 다르다: 인시던트 관리는 **서비스 복구(속도)**, 문제 관리는 **원인 제거(예방).**
- Atlassian의 관점: 두 실천법은 점점 더 **긴밀하게 얽히고(intertwined)** 있으며, 인시던트 사이의 시간에 팀이 문제 조사에 집중할 때 문제 관리는 가장 큰 가치를 낸다.

## 🔄 문제 관리 프로세스

문제 관리가 무겁고 사일로화된 별개 프로세스가 되면, 조직은 문제가 "죽으러 가는(go to die)" 백로그 더미(dumping ground)를 만들게 된다. Atlassian은 **문제 관리와 인시던트 관리를 서로 가깝게 붙이는(bring closer together)** 접근을 권장한다.

### 반응적(Reactive) vs 선제적(Proactive)

| 구분 | 반응적 문제 관리 | 선제적 문제 관리 |
|------|-----------------|-----------------|
| **시점** | 인시던트 발생 후 원인 조사 | 인시던트 발생 전 문제 탐지 |
| **한계/이점** | 근본 원인이 인시던트 이후 한참 뒤에야 조사되는 경향(사일로형의 병목 문제) | 인시던트를 유발하기 전에 문제를 찾아 고치거나 워크어라운드를 마련 |
| **권장 방식** | 인시던트와 통합하여, 장애 원인을 **해결과 동시에** 이해하는 접근 | 대응 모드가 아닐 때 서비스·성능에 가장 큰 영향을 주는 문제를 선제적으로 처리 |

> 예시: 소프트웨어 인시던트 해결 = 잘못된 코드(원인)를 식별 → 재발 방지를 위한 대체 코드 개발(수정). 원인 이해와 해결을 하나로 엮는 것이 선제적 접근.

### 단계별 프로세스

| 단계 | 명칭 | 활동 |
|------|------|------|
| 1️⃣ | **문제 탐지(Problem detection)** | 문제를 선제적으로 발견하여 수정하거나, 향후 인시던트 이전에 워크어라운드 식별. |
| 2️⃣ | **분류 및 우선순위 지정(Categorization & prioritization)** | 알려진 문제를 추적·평가하여 팀이 가장 관련성 높고 가치 있는 문제에 집중하도록 정리. |
| 3️⃣ | **조사 및 진단(Investigation & diagnosis)** | 문제의 근본적 기여 원인과 최선의 조치 방안(remediation)을 식별. |
| 4️⃣ | **알려진 오류 레코드 생성(Create a known error record)** | 근본 원인 + 워크어라운드를 문서화하여 KEDB에 저장. 문제가 인시던트로 번져도 다운타임을 줄임. |
| 5️⃣ | **워크어라운드 생성(필요 시)** | 문제를 즉시 제거할 수 없을 때, 영향을 줄이고 고객 대면 장애를 막는 임시 해결책 마련. |
| 6️⃣ | **문제 해결 및 종료(Resolve & close)** | 완전히 제거되어 더 이상 인시던트를 유발할 수 없는 상태가 된 문제를 종료. |

### ITIL 버전과 프로세스의 위상

- **ITIL V3:** 문제 관리를 권장 **프로세스(process)**로 명시.
- **ITIL V4:** 권장을 프로세스에서 제안된 **실천법(practice)**으로 전환. 조직이 자신의 고유한 상황과 필요에 맞게 스스로 프로세스를 정의하도록 권한을 부여.

## 👥 역할과 책임

> 모든 팀이 아래 모든 역할을 두는 것은 아니며, 반드시 영구적 직책도 아니다. 특정 문제를 해결하기 위해 모인 팀의 **역할 지정(designation)**으로 이해해야 한다. 효과적인 문제 관리는 팀원이 각 역할과 책임, 그리고 조사 중 누가 어떤 역할인지 명확히 아는 데서 나온다.

| 역할 | 책임 | 다른 명칭 |
|------|------|-----------|
| **문제 관리자 (Problem Manager)** | 특정 문제에 대한 전체 프로세스를 관리. 올바른 팀·도구·정보를 한데 모으는 등 문제 관리 노력의 모든 측면을 조율·지휘하고, 필요 시 하위 작업을 팀원에게 위임. | 문제 소유자(Problem owner), 주요 인시던트 관리자(Major Incident Manager) |
| **프로세스 소유자 (Process Owner)** | 팀의 문제 관리 프로세스 자체의 건전성과 성공을 책임. 프로세스의 진화·발전, 팀원 교육·온보딩을 감독. | 프로세스 관리자, 프로세스 코디네이터 |
| **서비스 소유자 (Service Owner)** | 서비스의 지속적 운영과 건전성을 정의. 변경·개선·계획된 다운타임·교육·문서화 등의 가치를 측정·보고. | 서비스 리더, 제품 관리자(Product manager) |
| **서비스 데스크 상담원 (Service Desk Agent)** | 서비스 데스크의 최전선 지원. 인시던트/문제를 가장 먼저 발견·보고하며, 여러 개별 인시던트가 하나의 더 큰 문제와 연관됨을 처음 알아차리는 경우가 많음. | 상담원, 서비스 에이전트, 지원 상담원, 헬프데스크 상담원, 서비스 데스크 분석가 |
| **테크 리드 (Tech Lead)** | 문제/인시던트를 겪는 영향받는 서비스에 정통한 개인. 주로 개발자·엔지니어로, 최근 코드 변경을 파고들어 기여 원인을 찾음. | 기술 리드, 주제 전문가(SME), 온콜 엔지니어, 개발자, SRE |
| **이해관계자 (Stakeholders)** | 프로세스에 직접 관여하지는 않지만 문제에 대한 상위 수준 정보가 필요한 이들. 인접 팀, 고객, 조직 리더 등. | 고객, 임원진, 벤더, 최종 사용자, 비즈니스 팀 |

## ✅ 모범 사례 (Best Practices)

가장 효과적인 팀은 **문제 관리와 인시던트 관리를 융합(blend)**한다. 문제 관리를 별개 실천으로 두면 문제 팀이 병목이 되거나, 통제 불가능한 외부 벤더 문제 같은 엉뚱한 곳에 집중하게 된다.

- 🚫 **반응적 RCA에만 의존하지 말 것** — 인시던트/문제 뒤에 근본 원인이 하나만 있는 경우는 드물다. 최고의 팀은 모든 잠재적 기여 요인을 **총체적으로** 고려하고 **비난 없는 분석(blameless analysis)**을 실천한다.
- 🗣️ **문제를 공유하는 열린 환경 조성** — 문제·인시던트 분석은 처벌이나 보복의 두려움 없이 사실을 공유하는 열린 대화여야 한다.
- 🎯 **중요 서비스에 집중** — 조직에 가장 큰 가치를 주는 서비스에 영향을 미치는 문제를 우선 해결.
- ❓ **질문하고 '5 Whys(다섯 번의 왜)'를 활용** — 토요타 생산 시스템의 설계자 오노 다이이치(Taiichi Ohno)가 창안한 기법. Atlassian Team Playbook에서 관련 플레이를 참고.
- 🔁 **지식을 전파(Spread the knowledge)** — 열린 팀은 동료·인접 팀이 배울 수 있도록 지식과 통찰을 공유.
- 📚 **학습하는 조직이 될 것** — 문제 관리는 종료일이 없다. 최고의 팀조차 인시던트를 겪는다. 진정한 월드클래스 팀은 프로세스를 끊임없이 반복·개선하여 문제의 영향을 줄인다.
- 📌 **후속 조치 추적(Track follow-up)** — 후속 조치를 관리하는 명확하고 표준화된 방식을 마련. 작업 우선순위 지정, 진행 추적, 인시던트-문제 연결을 지원하는 ITSM 소프트웨어를 활용.

## 📝 문제 기록/템플릿 구성 요소

템플릿을 사용하면 프로세스 일관성이 확보되고, 매번 문제에 대한 올바른 정보를 빠짐없이 수집할 수 있다. 아래는 Jira Service Management의 **문제(Problem) 티켓 커스텀 필드**로, 복사·수정하여 자체 문제 관리에 활용 가능하다.

| 필드 | 설명 |
|------|------|
| **Reporter (신고자)** | 요청을 제출한 사람. |
| **Assignee (담당자)** | 요청 처리를 배정받은 팀원. |
| **Priority (우선순위)** | 비즈니스 니즈·목표 대비 해결의 중요도. 종종 영향도(Impact)와 긴급도(Urgency)로 산정. |
| **Impact (영향도)** | 문제의 범위, 보통 SLA와 관련하여 측정. |
| **Urgency (긴급도)** | 비즈니스가 문제의 영향을 체감하기까지 남은 시간. |
| **Source (출처)** | 문제가 발생한 자산 또는 시스템. |
| **Investigation reason (조사 사유)** | 조사를 촉발한 트리거 (예: 반복 인시던트, 비정형 인시던트 등). |
| **Root cause (근본 원인)** | 문제와 관련된 인시던트들의 최초 원인. |
| **Workaround (임시 대응책)** | 서비스를 복구하기 위한 임시·알려진 해결책의 상세 설명. **JSM 지식 베이스에 문서화 권장.** |
| **Component(s) (구성 요소)** | 요청과 관련된 IT 인프라 세그먼트 (예: 결제 서비스, VPN 서버). 라벨링·분류·리포팅에 사용. |
| **Attachment (첨부)** | 요청에 추가된 파일·이미지. |
| **Summary (요약)** | 요청에 대한 짧은 설명. |
| **Description (설명)** | 요청에 대한 길고 상세한 설명. |
| **Linked Issues (연결 이슈)** | 요청에 영향을 주거나 받는 다른 요청 목록. 다른 Atlassian 제품 사용 시 개발 이슈도 포함 가능. |
| **Labels (라벨)** | 분류·조회를 위한 추가 커스텀 라벨. |
| **Request participants (요청 참여자)** | 요청에 참여하는 추가 고객 (타 팀, 벤더 등). |
| **Approvers (승인자)** | 요청 승인 책임자 (비즈니스·재무·기술 담당자). |
| **Organizations (조직)** | 요청 해결에 관심 있는 고객 그룹 목록. |
| **Pending reason (보류 사유)** | 문제가 진행되지 않는 이유를 나타내는 짧은 설명/코드. |
| **Product categorization (제품 분류)** | 요청이 영향을 미치는 IT 자산/시스템의 범주. |
| **Operational categorization (운영 분류)** | 요청 이행에 필요한 조치·기능의 범주. |

> **참고:** 위 필드 목록에서 "Summary/Description"이 두 번 등장하는 것은 원문 표기 그대로이며, 워크어라운드는 별도로 지식 베이스에 문서화하는 것이 권장된다.

## 📊 주요 지표 (Metrics)

Atlassian 문서는 별도의 KPI 표를 제시하지는 않으나, 본문 전반에서 강조하는 측정·관리 관점의 지표는 다음과 같다.

| 지표 관점 | 설명 |
|-----------|------|
| **해결 시간(Time to Resolution)** | 문제 분석 모범 사례 코드화로 단축되는 핵심 성과 지표. |
| **반복/재발 인시던트 수** | 반복 인시던트 감소는 문제 관리 효과성의 직접 척도이자 고객 신뢰의 원천. |
| **다운타임 비용** | Gartner 기준 시간당 30만 달러 이상 — 인시던트 회피의 재무적 가치 측정 기준. |
| **우선순위 = 영향도 × 긴급도** | 문제 티켓에서 우선순위 산정의 기본 공식. |
| **후속 조치 완료율(Follow-up tracking)** | 표준화된 방식으로 진행 상황을 추적해야 하는 관리 지표. |
| **서비스 가치 측정(Service Owner)** | 변경·개선·계획된 다운타임 등의 가치를 서비스 소유자가 측정·보고. |

## 🛠️ 도구 & Jira Service Management 연계

| 기능 | JSM의 지원 방식 |
|------|----------------|
| **인시던트-문제 연결** | 인시던트를 문제에 링크(link)하여 맥락 유지. |
| **근본 원인 분석 & 사후 검토** | RCA와 포스트인시던트 리뷰(post-incident review)를 지원. |
| **시정 조치 추적** | 시정 조치(corrective actions)를 완료까지 추적. |
| **협업 투명성** | 협업 도구 모음으로 팀이 더 높은 투명성과 맥락을 가지고 작업. |
| **지식 베이스** | 워크어라운드·해결책을 문서화하여 재사용. |
| **템플릿·커스텀 필드** | 문제 티켓 표준 필드(위 표)로 일관성 확보. |
| **자동화·AI·노코드 폼** | 수작업 프로세스 제거, 우선순위 지정·진행 추적 지원. |

> Atlassian의 핵심 메시지: **"인시던트 관리와 문제 관리가 손을 맞잡을 때(hand in hand)"** JSM의 가치가 극대화된다.

## 🔗 인시던트·변경 관리와의 관계

문제 관리는 인시던트 관리 및 기타 ITIL 실천법과 함께 작동하여 전체 ITSM 전략을 형성한다.

| 연계 실천법 | 관계 |
|-------------|------|
| **인시던트 관리(Incident Management)** | 문제 = 하나 이상 인시던트의 원인. 인시던트 관리는 복구(속도), 문제 관리는 원인 제거(예방). 점점 더 통합되는 추세. |
| **변경 관리(Change Management)** | 변경 = 서비스 중단·다운타임 없이 변경을 계획·추적·릴리스하는 프로세스. **변경이 중단/다운타임을 유발하면**, 그 변경은 인시던트·문제 관리 프로세스에서 분석된다. |
| **지식 관리(Knowledge Management)** | 일반 절차·워크어라운드에 대한 해결책·문서 저장소를 생성. 건강한 지식 관리는 더 빠른 인시던트 해결과 인시던트 감소를 가능케 함. |
| **서비스 요청 관리(Service Request Management)** | 사용자 요청(접근 권한, SW 개선, 정보 제공 등)을 처리. 인시던트와 구분이 어려울 수 있으며, ITIL V3(2007)부터 인시던트와 별도로 정의됨. |

## 🧭 요약 및 시사점

> **"인시던트는 서비스의 미래 신뢰성에 대한 계획되지 않은 투자다."** 효과적인 문제 관리는 인시던트를 유발하는 동인을 식별·제거하면서 동시에 가치 있는 서비스 개선을 배포한다.

1. **원인 제거가 목표다.** 서비스 복구(인시던트 종료)만으로는 문제가 남는다. 근본 원인과 기여 요인을 다뤄야 진짜로 끝난다.
2. **사일로를 피하라.** 별도의 무거운 문제 관리 팀은 병목·백로그 무덤이 된다. IT·보안·개발팀에 걸쳐 상시 실천하고, 인시던트 관리와 융합하라.
3. **선제적으로 전환하라.** 대응 모드가 아닐 때 가장 영향 큰 문제를 미리 처리하면 미래 인시던트를 예방한다.
4. **비난 없이, 열린 문화로.** 5 Whys 같은 기법과 blameless 분석으로 사실을 공유하는 학습 조직을 만들어라.
5. **도구로 지속성을 담보하라.** JSM 같은 ITSM 도구로 인시던트-문제 연결, KEDB, 후속 조치 추적을 표준화하라.
