# 서비스 요청 관리 (Service Request Management)

> **서비스 요청 관리(SRM)** 는 직원과 고객의 일상적인 요청(접근 권한, 장비, 정보 등)을 표준화된 워크플로우로 제출·분류·이행·소통하는 ITIL 프랙티스로, ITIL에서는 이를 **요청 이행(Request Fulfillment)** 프로세스가 담당한다.

---

## 📌 개요 (무엇을 & 왜)

IT 팀은 애플리케이션 접근, 소프트웨어 라이선스, 비밀번호 재설정, 신규 하드웨어 등 매우 다양한 요청을 받습니다. ITIL(Information Technology Infrastructure Library)은 이러한 요청을 **서비스 요청(Service Request)** 으로 분류합니다. 서비스 요청은 대부분 **반복적(recurring)** 이므로, 효율적인 IT 팀은 이를 **반복 가능한 표준 절차**로 처리합니다.

**핵심 3요소 (Atlassian 정의)**

| 구분 | 내용 |
|------|------|
| **무엇인가 (What)** | 요청이 제출·분류(triage)·이행·소통되는 방식을 규정한 표준화된 워크플로우 |
| **왜 중요한가 (Why)** | 잘 설계된 SRM은 사용자 만족도를 높이고, 대기 시간을 줄이며, 팀에게 명확한 수요 신호(demand signal)를 제공 |
| **어떻게 (How)** | Jira Service Management는 구성 가능한 요청 유형, 포털, 자동화를 제공해 대량 요청을 효율적·투명하게 처리 |

> 💡 **핵심 관점**: 서비스 요청 관리는 "사용자에게 새로운 무언가(접근 권한, 새 폰, 정보)를 제공하도록 **활성화(enabling)**"하는 것이 본질입니다. 요청 이행은 사용자가 (1) 이용 가능한 서비스를 보고, (2) 요청 방법을 이해하며, (3) 처리 소요 시간에 대한 기대치를 갖도록 돕습니다.

**왜 별도 워크스트림으로 다루는가?**
- 서비스 요청은 대개 **저위험(low risk)** 이므로 신속 처리하거나 자동화할 수 있음 (예: 신규 직원의 SW 접근 요청은 사전 승인 후 자동 부여 가능)
- 대량 요청이 발생하는 조직에서는 **별도 레코드 유형**으로 기록·관리하는 것이 현명함
- 인시던트/변경/문제 요청과 분리하면 자원 배분을 최적화하고 워크플로우 복잡도를 낮출 수 있음

## 🧩 핵심 개념 및 용어 정리

ITIL의 네 가지 핵심 작업 유형은 자주 혼동되므로 명확히 구분해야 합니다.

| 용어 | 정의 | 예시 |
|------|------|------|
| **서비스 요청 (Service Request)** | 새로운 무언가를 제공받기 위한 사용자의 공식 요청 | "새 맥북이 필요해요." |
| **인시던트 (Incident)** | 서비스를 중단·저하시키는 계획되지 않은 이벤트로 긴급 대응 필요 | "웹사이트가 다운됐어요!" |
| **문제 (Problem)** | 반복적·예방 가능한 인시던트의 근본 원인 | "그 애플리케이션 오류가 또 발생했어요!" |
| **변경 (Change)** | IT 서비스에 영향을 줄 수 있는 것의 추가·수정·제거 (서비스 요청과 연계 가능) | "데이터베이스를 업그레이드해야 해요!" |

**주요 용어**

- **요청 이행 (Request Fulfillment)**: 고객의 서비스 요청을 해결하는 프로세스로, 모든 서비스 요청의 전체 생애주기를 관리
- **서비스 데스크 (Service Desk)**: ITIL 4 정의로 "서비스 제공자와 사용자 간의 단일 접점(single point of contact)". 인시던트와 서비스 요청을 관리하고 사용자와의 커뮤니케이션을 담당
- **서비스 카탈로그 (Service Catalog)**: 배포 가능한 라이브 IT 서비스 목록 ("스토어프론트" 역할)
- **서비스 포트폴리오 (Service Portfolio)**: IT 조직의 과거·현재·미래 서비스 전체 기록
- **요청 유형 (Request Type)**: 포털에서 사용자가 보게 되는 제출 옵션 (예: "IT 도움 받기", "새 계정 요청")
- **SLA (Service Level Agreement)**: 서비스 제공자와 고객 간의 평이한 언어로 된 합의, 응답·해결 시간 목표 규정
- **Shift Left (좌측 이동)**: 요청 이행을 최전선(고객)에 최대한 가깝게 옮기는 것

## 🔄 서비스 요청 프로세스 (단계별)

### 간략형 (ITIL 기반 4단계)

1. 고객이 서비스 포털 또는 이메일을 통해 도움을 요청
2. IT 서비스 팀이 사전 정의된 승인·자격 프로세스에 따라 요청을 평가 (필요 시 재무·비즈니스 승인 요청)
3. 서비스 데스크 상담원이 요청을 이행하거나, 이행 가능한 담당자에게 전달
4. 요청 해결 후 상담원이 티켓을 종료하고, 고객에게 만족 여부를 확인

### 상세형 (7단계 워크플로우)

| 단계 | 무슨 일이 일어나는가 | 왜 중요한가 |
|------|----------------------|-------------|
| **① 제출 (Submission)** | 요청자가 포털·헬프센터·양식을 통해 요청 제출 | 접수를 표준화하고 핵심 정보를 사전 확보 |
| **② 검증 (Validation)** | 완전성·정확성·자격 여부 확인 | 지연을 줄이고 불완전한 요청의 진행 방지 |
| **③ 분류 및 라우팅 (Categorization & Routing)** | 적절한 팀·큐·워크플로우에 배정 | 요청이 올바른 담당자에게 더 빨리 도달 |
| **④ 승인 (Approval)** | 필요 시 적절한 승인자에게 전달 | 비용·보안·규정 준수 관련 감독 추가 |
| **⑤ 이행 (Fulfillment)** | 배정된 팀이 요청 작업 완료 | 요청자가 필요로 하는 서비스 제공 |
| **⑥ 검증 (Verification)** | 요청자·팀이 올바른 완료 여부 확인 | 종료 전 품질과 만족도 보장 |
| **⑦ 종료 (Closure)** | 요청 종료 및 기록 | 리포팅 및 향후 프로세스 개선 지원 |

> ⚠️ **표준화가 핵심**: 요청을 포착·이행하는 방식에는 변형이 있을 수 있으나, 전체 서비스 품질과 효율을 높이려면 **표준화를 추진하는 데 집중**해야 합니다.

**JSM에서의 서비스 요청 제출 흐름 (예시)**
1. Jira Service Management 포털/헬프센터 열기 (서비스 카탈로그로 옵션 정리, 포털 그룹으로 분류)
2. 요청 유형 선택 (예: "IT 도움 받기", "새 계정 요청")
3. 맞춤형 요청 양식 작성 (필드 예: 접근 수준, 기기 유형, 시작일, 비즈니스 사유)
4. 요청 제출 → JSM 내 작업 항목(work item)으로 생성
5. 상태 업데이트 추적 및 후속 질문 응답 (실시간 가시성)
6. 서비스 팀이 큐에서 요청을 분류(triage) 및 배정

**자주 발생하는 프로세스 문제**
- 혼란스럽거나 잘못 명명된 요청 유형 → 사용자가 올바른 유형 선택 곤란
- 요청 유형에 대한 제한된 가시성 → 사용자 혼란
- 느리거나 일관성 없는 라우팅 → 해결 시간 증가
- 너무 많은 채널로 들어오는 요청 → 우선순위 지정·해결 어려움

## 👥 역할과 책임

| 역할 | 책임 |
|------|------|
| **요청자 (Customer/Requester)** | 포털·이메일·양식을 통해 요청 제출, 후속 정보 제공, 상태 추적 |
| **서비스 데스크 상담원 (Agent)** | 요청 접수·분류·이행 또는 전달, 티켓 종료, 만족도 확인 |
| **프로세스 오너 (Process Owner)** | 요청 제공 항목(request offering)을 시간에 따라 관리, 요구사항 문서화 |
| **승인자 (Approver)** | 비용·보안·규정 관련 요청 승인 (예: 관리자, 예산 담당자, 보안 팀) |
| **이행 팀 (Fulfillment Team)** | 실제 작업 수행 (IT, HR, 재무, 시설 등 다양한 내부 팀 가능) |
| **IT 관리자 (IT Manager)** | 기능 영역별 역할 분담, 수요 기반 자원 배분, 전략 수립 |

> 🤝 **협업적 접근 권장**: 전통적인 계층형(tiered) 지원팀은 에스컬레이션으로 요청을 관리하지만, Atlassian은 모든 팀원이 고객에 가까이 다가가 응답하는 **협업적 스워밍(swarming)** 접근을 권장합니다. Slack 등에서 함께 이슈를 처리하면 모두가 학습 기회를 얻습니다. 주간 회고(retro)와 헬스 모니터로 지속적 개선을 도모하세요.

## ✅ 모범 사례 (Best Practices)

### SRM 우선순위 3원칙
1. **지원팀을 지원하라 (Support the support team)**: 최전선 팀의 웰빙과 성장에 집중. 협업·학습 중심 문화로 번아웃 방지
2. **Shift Left (좌측 이동)**: 요청 이행을 고객·최전선에 최대한 가깝게. 검색 가능한 지식베이스로 티켓 편향(deflection), 맞춤형 접수 양식으로 왕복 대화 축소, 단일 셀프서비스 포털 중앙화
3. **자동화 활용 (Leverage automation)**: 반복 작업 제거, 후속 커뮤니케이션 자동화, 정형 응답(canned response), 적절한 팀으로 자동 라우팅
4. **확장 대비 (Be ready to scale up)**: 서비스 카탈로그로 라이브 서비스 정보 제공, 개발자 없이 신속 배포

### 서비스 요청 8가지 실행 팁
1. **가장 흔하고 단순하며 쉽게 이행되는 요청부터 시작** — 즉각적 가치 제공 + 팀 학습
2. **모든 요구사항 문서화** — 질문 필드, 승인 프로세스, 이행 절차, 이행 팀, 프로세스 오너, SLA, 리포팅 등
3. **접수 시 필요한 데이터를 포착하되, 너무 많은 질문으로 고객을 압도하지 말 것**
4. **가능한 곳에서 승인 프로세스 표준화·자동화** (예: 모니터 요청은 사전 승인, SW 요청은 관리자 승인)
5. **이행 프로세스 검토** — 책임 팀 식별, 특수 요구사항 확인, 자동화로 간소화
6. **지식베이스에 필요한 정보 식별** — FAQ로 답할 수 있으면 요청 제공 항목 계획에 포함 (셀프서비스 편향)
7. **SLA 검토** — 적절한 측정과 알림으로 적시 이행 보장
8. **리포팅·지표 식별** — CSAT, 응답 시간(time to response), 해결 시간(time to resolution), 종료 시간(time to close) 추적 시작

## 📊 주요 지표 (KPI/Metrics)

| 지표 | 정의/설명 | 참고 수치·사례 |
|------|-----------|----------------|
| **CSAT (고객 만족도)** | 고객 만족도 측정 | FCR 1% 개선 시 CSAT 1% 개선 (SQM Group); 동일 이슈 재문의 시 CSAT 평균 15% 하락 |
| **FCR (First Call Resolution)** | 첫 접촉 시 고객 요구를 완전히 충족하는 비율 | KCS 도입 조직은 FCR·1차 해결률이 최대 15% 더 높음 (HDI) |
| **재오픈율 (Reopen Rate)** | 종료된 티켓이 다시 열리는 비율 | FCR 정확도의 "견제와 균형" 장치. 재오픈 측정 창은 최초 통화 후 24~48시간 권장 |
| **MTTR** | 4가지 의미(resolve/respond/repair/recovery) | 단일 지표가 아님에 주의 |
| **Mean time to resolve** | 티켓 오픈~종료 평균 경과 시간 | — |
| **Mean time to respond** | 신규 티켓에 대한 평균 응답 시간 | — |
| **Mean time to recovery** | 문제 탐지~완화~해결 시간 | DORA 연구의 핵심 DevOps 지표 |
| **Uptime (가동시간)** | 시스템이 이용 가능·작동하는 시간 비율 | 예: 99.9% / 99.95% |
| **Cost Per Ticket (티켓당 비용)** | 서비스 데스크 운영비 ÷ 티켓 볼륨 | 서비스 데스크 효율 척도 |
| **Time to first response** | 최초 응답까지 시간 | FCR 보조 지표로 병행 권장 |
| **SLA 준수율** | 충족된 SLA 비율 | — |

**DevOps 도입 성과 지표 (참고)**: 고객 만족도 45%↑, 직원 생산성 43%↑, 결함률 41%↑ 개선, IT 관련 비용 38%↓ / 기업의 4/5가 DevOps 원칙 일부 사용 / 변경·릴리스가 인시던트의 85~87% 유발 (Gartner)

> 📏 **지표 선택 원칙**: FCR은 하나의 측정치일 뿐, 고객 만족이나 IT 효과성의 종합 지표가 아닙니다. FCR을 **재오픈율·CSAT와 함께** 사용하세요. 또한 상충하는 목표(예: FCR 85% + 5분 내 해결/에스컬레이션)를 설정하지 마세요. **최우선 목표는 FCR이 아니라, 최선의 방식으로 문제를 해결하는 것**입니다.

**리포팅 4대 모범 사례**: ① IT 성과 목표를 비즈니스 목표와 정렬, ② 이해관계자별로 다른 지표 공유(현장 관리자 vs CTO), ③ 시각화로 이해 용이하게(트렌드 리포트 활용), ④ 스코어카드 유지(빨강/노랑/초록 신호등, 정직하게 평가)

## 🛠️ 도구 & Jira Service Management 연계

### JSM 커스터마이징 6가지 방법
1. **요청 유형 구성** — 기본 유형 수정 또는 신규 추가, 드래그앤드롭 편집기로 워크플로우 매핑
2. **요청 유형 정리** — 7개 이상이면 그룹화, 아이콘·설명 편집
3. **동적 요청 양식 구축** — ProForma로 조건부 로직 기반 스마트 폼/체크리스트, 필드 동적 표시/숨김
4. **헬프센터에 브랜딩 적용** — 로그인 메시지·공지·룩앤필·언어 커스터마이징, Refined 앱으로 고급 브랜딩
5. **지식베이스 연결** — Confluence 통합으로 셀프서비스 강화, 스마트 검색, 지식 리포트·스코어링
6. **요청 위젯 추가** — 임베드 가능 위젯으로 어디서나 요청, 큐로 자동 유입

### 티켓팅 시스템 & 채널
- **IT 티켓팅 시스템**: 티켓 생성 → 분류/우선순위 → 담당자 배정 → 해결 → 종료. 자동화로 조직화·소통·데이터 기반 인사이트 강화
- **대화형 티켓팅 (Conversational Ticketing)**: Slack/Teams에서 실시간 지원. 양방향 동기화가 필수
- **이메일 지원 전환**: 발신자→요청자(reporter), 제목→요약(summary), 본문→설명(description)으로 매핑. 고객은 변화를 눈치채지 못한 채 기존 이메일 주소 계속 사용
- **가상 에이전트 (Virtual Agent)**: Rovo 기반 AI가 tier-1 요청(비밀번호 재설정, 접근 부여, 상태 확인, 정책 조회) 편향 처리, NLP로 맥락 해석, 인간 상담원에게 전체 대화 이력과 함께 인계

### 참고 도구 (help desk 페이지에서 언급된 경쟁·대안)
Jira Service Management(통합 ITSM 1위로 소개), Zendesk(옴니채널, 1,500+ 앱), Freshservice(ITIL 정렬, Freddy AI), Help Scout(단순·협업), Zoho Desk(가성비, Zia AI), SolarWinds Service Desk(자산관리·CMDB), Freshdesk(멀티채널 고객지원), HubSpot Service Hub(고객 포털·CRM 통합)

## 🔗 다른 ITIL 프랙티스와의 관계

ITSM은 ITIL이 정의한 여러 핵심 프로세스로 구성되며, SRM은 그중 하나입니다.

| 프랙티스 | SRM과의 관계 |
|----------|--------------|
| **인시던트 관리 (Incident)** | SRM은 "새로운 것 제공", 인시던트는 "중단된 서비스 복구". 별도 워크스트림으로 분리 권장. 서비스 카탈로그가 요청 유형 급증을 통해 인시던트를 조기 표면화 |
| **문제 관리 (Problem)** | 반복 인시던트의 근본 원인 해결. SRM보다 상위/예방적 활동 |
| **변경 관리 (Change)** | 서비스 요청이 변경과 연계 가능 (예: SW 프로비저닝 요청이 변경 관리 워크플로우 실행). 서비스 카탈로그+서비스 데스크가 고위험 서비스에 적절한 관리 수준 보장 |
| **지식 관리 (Knowledge)** | 지식베이스로 요청 편향(deflection), FCR 향상. KCS(Knowledge-Centered Service) 도입 시 1차 해결률 최대 15%↑ |
| **IT 자산 관리 (ITAM)** | 기본 IT 지원 범위 밖의 ITSM 활동. 하드웨어 요청 등과 연계 |

**계층 구조 이해 (help desk vs service desk vs ITSM)**
- **ITSM**: IT 서비스의 전체 end-to-end 제공을 관리하는 최상위 개념 (서비스 요청·지식·자산·인시던트·문제·변경 관리 포함)
- **서비스 데스크**: 단일 접점. 서비스 요청+인시던트 관리, 지식관리, 셀프서비스, 리포팅 포함. 전략적(strategic)
- **헬프 데스크**: 주로 break-fix(인시던트 관리)에 집중. 전술적(tactical). 서비스 데스크의 하위 집합

> 🧭 서비스 데스크는 ITIL 틀에서 태어난 헬프 데스크의 **진화형**입니다. 헬프 데스크는 "도움(help)"을, 서비스 데스크는 "서비스(service)"를 제공합니다. HDI 조사에 따르면 헬프 데스크·서비스 데스크의 41%가 다른 이름으로 불립니다.

**ITIL vs DevOps**: 경쟁이 아닌 **상호 보완** 관계. ITIL은 운영·지원·거버넌스 모범 사례를, DevOps는 지속적 배포·무비난(blameless) 문화·협업 도구·애자일 관행을 제공. 고성과 조직은 둘 다 활용합니다.

## 📚 하위 주제 요약

### 서비스 요청 관리 (개요)
서비스 요청 관리는 일상적 요청을 처리하는 프로세스로, ITIL의 요청 이행 프로세스가 담당. 인시던트/문제/변경과 구별되며 별도 워크스트림 권장. 고객 중심·지식 중심·자동화 간소화가 핵심 우선순위.

### 서비스 요청 프로세스
제출→검증→분류/라우팅→승인→이행→검증→종료의 7단계 표준 워크플로우. 명확한 프로세스는 요청 접수처를 명확히 하고 접수 품질을 높이며 왕복·수동 분류를 줄이고 승인/핸드오프를 지원. JSM에서 포털·요청 유형·양식·큐·승인 워크플로우로 구현.

### 대화형 티켓팅 (Conversational Ticketing)
Slack/Teams 등 채팅 플랫폼에서 실시간으로 이뤄지는 지원. Fortune 500의 65%가 Slack, 91%가 Teams 사용; 66%가 둘 다 사용. Slack 12M DAU, Teams 75M 사용자. **Slack은 티켓 처리 시간을 절반 이상 단축**, **Strava는 응답 시간 90% 개선**. 라이브 챗은 타 방식 대비 **50% 비용 절감**. 도전 과제(빠른 피드 추적, 지표 추적)는 JSM 같은 도구로 해결. 모범 사례: 단순하게 유지, 조직화, 데이터 아카이브, **양방향 동기화**, 흔한 이슈 셀프서비스 제공.

### 첫 통화 해결 (First Call Resolution, FCR)
IT 팀이 첫 접촉에서 고객 요구를 완전히 충족하는 능력. **FCR 1% 개선 = CSAT 1% 개선; 재문의 시 CSAT 15% 하락** (SQM Group). 모범 사례: 측정 대상을 명확·일관되게 정의, 고객이 확인해야 해결로 표시, 재오픈율 추적(24~48시간 창), 전체 고객 여정 관찰, 지식베이스 도입(KCS로 최대 15%↑), 상충 목표 회피. **FCR은 최우선 목표가 아니며** 최선의 문제 해결이 우선.

### 헬프 데스크 (Help Desk)
고객과 직원의 첫 접점. IT 지원 티켓을 로그·추적·해결. 유형: IT 지원 / 고객 서비스 / 클라우드 기반 / 온프레미스. 2026년 선택 기준: 확장성, 통합, 사용 편의성, 분석 역량. 무료 체험으로 실제 팀과 테스트 권장.

### 헬프 데스크 vs 서비스 데스크 vs ITSM
ITSM(전체 서비스 생애주기) ⊃ 서비스 데스크(요청+인시던트+지식+셀프서비스+리포팅, 전략적) ⊃ 헬프 데스크(break-fix 중심, 전술적). 서비스 데스크 ITIL 4 정의: "서비스 제공자와 사용자 간 단일 접점". 헬프 데스크의 41%가 다른 이름으로 불림(HDI).

### 서비스 데스크 구축 방법 (6가지 모범 사례)
Atlassian 자체 경험 기반(2010년 230명→수천 명 성장). ① 서비스 데스크 SW를 최대한 활용, ② IT 팀을 "만능 처리반"으로 취급 중단(사무 엔지니어링/워크플레이스 테크/글로벌 데스크로 분화), ③ 고객 포털 구축(go/ithelp), ④ SLA를 스마트하게 설정, ⑤ 셀프서비스 촉진(**72%가 셀프서비스 선호**), ⑥ 큰 그림을 보고 진척 측정(인시던트 예방 중심).

### JSM 커스터마이징 방법 (6가지)
요청 유형 구성·정리, 동적 양식(ProForma), 헬프센터 브랜딩(Refined 앱), 지식베이스 연결(Confluence), 요청 위젯. 유연성으로 IT·HR·법무·마케팅 등 다양한 팀 지원.

### IT 지원을 DevOps 방식으로 운영하기
DevOps는 개발·운영 사일로를 협업으로 연결. 성과: CSAT 45%↑, 생산성 43%↑, 결함률 41%↑, IT 비용 38%↓. 모범 사례: 문화 변화 우선, 가능한 곳 자동화(워크플로우/지식/에스컬레이션), 중요 지표 추적(MTBF·MTTR·MTTF·MTTA), 공유(오너십·책임·성공) 우선. 변경이 인시던트의 85~87% 유발(Gartner). SRE 접근으로 개발·운영 목표 충돌 해소.

### IT 지표 및 리포팅
IT 지표 = IT 리더가 IT 비즈니스를 관리하는 정량 측정치. 공통 지표: MTTR, Uptime, Cost Per Ticket, CSAT, SLA. 4대 모범 사례: 비즈니스 목표 정렬, 이해관계자별 맞춤 공유, 시각화, 스코어카드 유지.

### IT 지원 (IT Support)
기술 이슈를 해결하는 서비스·전문가. 유형: 사내/아웃소싱/원격/온사이트. 3계층(L1 셀프헬프·지식베이스, L2 헬프데스크·서비스데스크, L3 심층). 이점: 다운타임 최소화, 데이터 보안 강화, UX 개선, 확장성 지원. SW 핵심 기능: 티켓팅, 지식 관리, 커뮤니케이션 도구, 성과 지표.

### IT 티켓팅 시스템
IT 티켓 = 서비스 요청을 추적·관리하는 지원 티켓. 생성→분류/우선순위→배정→해결→종료. 이점: 조직화 향상, 커뮤니케이션 강화, 데이터 기반 인사이트. 자동화가 핵심 기능. ITSM의 한 구성요소로, ITSM SW와 통합되는 티켓팅 시스템 선택이 중요.

### 서비스 카탈로그 (Service Catalog)
회사 서비스의 포괄적 목록. 비즈니스 카탈로그("스토어프론트", 최종 사용자용) vs 기술 카탈로그(내부용). 서비스 포트폴리오(과거·현재·미래 전체 기록)와 구별. 구성요소: 서비스 오퍼링·옵션, 서비스 관계. 관리: 역할·책임, 거버넌스, 성과 지표(사용량·SLA·CSAT). 변경·인시던트 관리와 연계되어 ITSM 전 단계에 관여.

### 서비스 데스크 / IT 서비스 포털
IT 서비스 포털 = 고객이 도움을 구하는 온라인 허브(셀프서비스 센터). JSM에서는 각 서비스 데스크에 대응하며 헬프센터가 모든 포털을 보유. 이점: UX 향상, 일관된 지원, 더 나은 인시던트 관리, 개선된 커뮤니케이션. 핵심 기능: 셀프서비스, 서비스 요청 관리, 리포팅·분석.

### SLA (Service Level Agreement)
서비스 제공자와 고객 간 평이한 언어의 합의(예: 99.9% 가동시간, 24시간 내 응답). 핵심 구성: 제공 서비스, 성과 지표, 역할·책임, 구제·페널티. 유형: 고객 기반 / 서비스 기반 / 다단계(multi-level). 지표: 가동시간, 응답 시간, 해결 시간, 품질 지표. 모범 사례: 고객 응답 대기 중 시간 측정 중단, 상담원 경험 고려(단순 명명), 대형 SLA 분할, 우선순위별 목표 차등, 24/7 vs 업무시간 구분, 루틴 자동화, AI 활용(Rovo).

### 이메일 지원에서 전환하기
이메일은 사라지지 않지만 과도한 왕복·넘치는 받은편지함 등 숨은 시간 낭비 존재. JSM으로 전환: 이메일→티켓 매핑(고객은 눈치채지 못함), 요청 유형·워크플로우 커스터마이징, SLA 설정, 팀 협업·중앙화 소통. CSAT·해결 시간 단축으로 개선 측정.

### 가상 에이전트 (Virtual Agent)
ML·NLP 기반의 AI로 단순 챗봇 이상. tier-1 지원 자동화, 흔한 이슈 편향, 인간 상담원 직접 지원(요청 요약·심각도 분류·맥락 표면화). 가상 비서(virtual assistant, 원격 근무 인력)와 구별. 기능: NLP, 음성 비서, 대화형 분석, 다국어 지원. 활용: 고객서비스·영업/마케팅·재무·HR. JSM에서는 Rovo 기반으로 Slack/Teams/포털에서 작동, Confluence 지식베이스 활용, Service Collection으로 IT·HR·고객지원 통합.

## 🧭 요약 및 시사점

- **서비스 요청 관리는 "활성화(enablement)"의 실천이다.** 직원·고객이 필요한 IT 서비스에 접근하도록 돕는 것이 목적이며, 인시던트/문제/변경과 명확히 구별해 **별도 워크스트림**으로 다루는 것이 자원 최적화의 핵심입니다.
- **표준화 + 자동화 + 셀프서비스**가 3대 성공 동력입니다. 저위험 반복 요청은 사전 승인·자동 이행하고, 지식베이스로 티켓을 편향(deflection)하며, Shift Left로 최전선에서 해결하세요.
- **지표는 균형 있게.** FCR·CSAT·재오픈율·MTTR·Cost Per Ticket을 함께 보고, 상충 목표를 피하며, 지표를 비즈니스 목표와 정렬하고 이해관계자별로 다르게 리포팅하세요.
- **도구는 진화 경로를 지원해야 한다.** 헬프 데스크(전술) → 서비스 데스크(전략) → 완전한 ITSM으로 성숙도에 따라 확장 가능한 단일 플랫폼(예: JSM)이 유리합니다.
- **AI·대화형·옴니채널이 미래다.** 대화형 티켓팅(Slack/Teams)과 Rovo 기반 가상 에이전트는 tier-1 편향과 해결 시간 단축(Slack 50%+, Strava 90%)의 실증적 성과를 보여줍니다.
- **문화가 도구를 앞선다.** DevOps·협업·무비난 문화, 지원팀의 웰빙, 지속적 개선(회고·헬스 모니터)이 뒷받침될 때 도구의 효과가 극대화됩니다.
