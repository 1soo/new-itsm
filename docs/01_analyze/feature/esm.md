## 3. 기능 상세

### FEAT-ESM-001 · 부서별 카탈로그 관리

- **설명**: 프로세스 오너가 담당 부서를 지정한 서비스 카탈로그 항목을 관리한다.
- **관련 요구사항**: REQ-ESM-001
- **입력**: 담당 부서(HR/법무/시설/재무), 이름, 설명, 양식 필드
- **출력**: 등록된 카탈로그 항목
- **처리 흐름**:
  1. 담당 부서·필수 필드 검증
  2. 카탈로그 항목 저장
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 프로세스 오너가 담당 부서를 포함해 카탈로그 항목을 생성하면, 시스템은 이를 등록해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 담당 부서가 지정되지 않으면, **THEN** 시스템은 저장을 거부해야 한다.

### FEAT-ESM-002 · 부서 요청 제출·처리

- **설명**: 요청자가 부서 포털에서 요청을 제출하고 담당 부서 큐로 라우팅되어 처리된다.
- **관련 요구사항**: REQ-ESM-002
- **입력**: 카탈로그 항목 선택, 요청 양식 값
- **출력**: 생성된 요청, 상태 이력
- **처리 흐름**:
  1. 요청 생성
  2. 담당 부서 큐로 라우팅
  3. 처리자 상태 갱신 시 요청자에게 반영
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 요청자가 요청을 제출하면, 시스템은 담당 부서 큐로 라우팅해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 필수 양식 필드가 누락되면, **THEN** 시스템은 제출을 거부해야 한다.

### FEAT-ESM-003 · HR 케이스 관리

- **설명**: HR 담당자가 민감 이슈를 4단계 상태로 추적한다.
- **관련 요구사항**: REQ-ESM-003
- **입력**: 케이스 내용, 상태 전이 요청
- **출력**: 케이스 상태·이력
- **처리 흐름**:
  1. 케이스 접수(Intake)
  2. 기록(Documentation)
  3. 조사(Investigation)
  4. 해결(Resolution)
- **인수 기준 (EARS)**:
  - (Ubiquitous) 시스템은 HR 케이스를 접수·기록·조사·해결 상태로 관리해야 한다.
  - (Event-driven) **WHEN** 상태를 전이하면, 시스템은 이력을 기록해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 정의되지 않은 상태로 전이하려 하면, **THEN** 시스템은 거부해야 한다.

### FEAT-ESM-004 · HR 케이스 접근 제한

- **설명**: HR 역할 또는 SYSTEM_ADMIN을 가진 사용자만 HR 케이스에 접근할 수 있다.
- **관련 요구사항**: REQ-ESM-004
- **입력**: 요청 사용자 역할, 케이스 ID
- **출력**: 허용/거부 응답
- **처리 흐름**:
  1. 사용자 역할 확인
  2. HR 역할도 SYSTEM_ADMIN도 아니면 차단
- **인수 기준 (EARS)**:
  - (Ubiquitous) 시스템은 HR 케이스 접근을 HR 역할과 SYSTEM_ADMIN으로 제한해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** HR 역할이 아니고 SYSTEM_ADMIN도 아닌 사용자가 접근하면, **THEN** 시스템은 403을 반환해야 한다.

### FEAT-ESM-005 · 온보딩 체크리스트

- **설명**: 신규 입사 요청 시 체크리스트를 생성하고 관련 부서에 하위 작업을 배정한다.
- **관련 요구사항**: REQ-ESM-005
- **입력**: 온보딩 요청, 체크리스트 템플릿
- **출력**: 생성된 체크리스트와 하위 작업(부서별)
- **처리 흐름**:
  1. 온보딩 요청 접수
  2. 체크리스트 템플릿 적용
  3. IT/시설/HR 등 하위 작업 배정
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 신규 입사 요청이 제출되면, 시스템은 온보딩 체크리스트를 생성하고 관련 부서에 하위 작업을 배정해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 체크리스트 템플릿이 정의되지 않으면, **THEN** 시스템은 생성을 거부해야 한다.

### FEAT-ESM-006 · 오프보딩 체크리스트

- **설명**: 퇴사 요청 시 체크리스트를 생성하고 자산 회수·접근 권한 해제 작업을 배정한다.
- **관련 요구사항**: REQ-ESM-006
- **입력**: 오프보딩 요청, 대상자 보유 자산 목록
- **출력**: 생성된 체크리스트와 자산 회수 하위 작업
- **처리 흐름**:
  1. 오프보딩 요청 접수
  2. 자산(ITAM) 도메인에서 대상자 보유 자산 조회
  3. 자산 회수·계정 해제 하위 작업 생성 및 연결
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 퇴사 요청이 제출되면, 시스템은 오프보딩 체크리스트를 생성하고 자산 회수·접근 권한 해제 하위 작업을 배정해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 대상자에게 연결된 자산이 없으면, **THEN** 시스템은 자산 회수 하위 작업 없이 체크리스트를 생성해야 한다.

### FEAT-ESM-007 · 체크리스트 진행 추적

- **설명**: 온보딩/오프보딩 체크리스트의 하위 작업 상태와 전체 완료 여부를 표시한다.
- **관련 요구사항**: REQ-ESM-007
- **입력**: 체크리스트 ID
- **출력**: 하위 작업 상태 목록, 전체 완료 여부
- **처리 흐름**:
  1. 하위 작업 상태 조회
  2. 전체 완료 여부 계산
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 모든 하위 작업이 완료되면, 시스템은 체크리스트를 완료로 표시해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 하위 작업 중 하나라도 미완료면, **THEN** 시스템은 체크리스트를 진행중으로 표시해야 한다.

### FEAT-ESM-008 · ESM 지표 리포팅

- **설명**: 부서별 처리 건수·평균 처리시간·온보딩/오프보딩 완료율을 집계한다.
- **관련 요구사항**: REQ-ESM-008
- **입력**: 조회 기간·부서 필터
- **출력**: 부서별 처리량, 평균 처리시간, 완료율
- **처리 흐름**:
  1. 부서별 요청/케이스/체크리스트 데이터 집계
  2. 지표 계산·반환
- **인수 기준 (EARS)**:
  - (Ubiquitous) 시스템은 부서별 처리 건수·평균 처리시간·완료율을 조회할 수 있어야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 해당 기간에 데이터가 없으면, **THEN** 시스템은 빈 결과를 반환해야 한다.
