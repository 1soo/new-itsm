# 기능 명세서 — 인시던트 관리 (Incident Management)

## 3. 기능 상세

### FEAT-INC-001 · 인시던트 등록

- **설명**: 사용자가 인시던트를 생성하고 핵심 필드를 기입한다.
- **관련 요구사항**: REQ-INC-001
- **입력**: 요약, 설명, 심각도, 영향 서비스, 영향 제품
- **출력**: 생성된 인시던트(식별키·초기 상태 신규)
- **처리 흐름**:
  1. 입력값 유효성 검사
  2. 인시던트 생성(상태=신규)
  3. 식별키 반환
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 필수 필드를 포함해 등록하면, 시스템은 인시던트를 생성해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 요약·심각도가 누락되면, **THEN** 시스템은 400 오류를 반환해야 한다.

### FEAT-INC-002 · 심각도·우선순위 관리

- **설명**: 인시던트에 SEV 등급과 우선순위를 부여·변경한다.
- **관련 요구사항**: REQ-INC-002
- **입력**: 인시던트 ID, 심각도(SEV 1~3), 우선순위
- **출력**: 갱신된 심각도·우선순위, 변경 이력
- **처리 흐름**:
  1. 권한 확인
  2. 값 갱신 및 이력 기록
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 심각도/우선순위를 변경하면, 시스템은 값을 갱신하고 이력에 남겨야 한다.
  - (Ubiquitous) 시스템은 심각도와 우선순위를 독립적으로 관리해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 정의되지 않은 심각도 값을 입력하면, **THEN** 시스템은 이를 거부해야 한다.

### FEAT-INC-003 · 상태 전이

- **설명**: 인시던트를 정의된 라이프사이클에 따라 전이한다.
- **관련 요구사항**: REQ-INC-003
- **입력**: 인시던트 ID, 목표 상태
- **출력**: 갱신된 상태
- **처리 흐름**:
  1. 현재 상태에서 목표 상태로의 전이 허용 여부 확인
  2. 전이·이력 기록
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 허용된 전이를 요청하면, 시스템은 상태를 갱신해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 허용되지 않은 전이를 시도하면, **THEN** 시스템은 이를 거부해야 한다.

### FEAT-INC-004 · 역할 배정

- **설명**: Incident Manager가 대응 역할을 배정한다.
- **관련 요구사항**: REQ-INC-004
- **입력**: 인시던트 ID, 대상 사용자, 역할(Tech Lead·Comms·Scribe 등)
- **출력**: 배정 결과
- **처리 흐름**:
  1. Incident Manager 권한 확인
  2. 역할 배정·표시
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** IM이 역할을 배정하면, 시스템은 배정을 기록·표시해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** IM 권한이 없으면, **THEN** 시스템은 403을 반환해야 한다.

### FEAT-INC-005 · 에스컬레이션

- **설명**: 인시던트를 상위/전문 담당자에게 이관한다.
- **관련 요구사항**: REQ-INC-005
- **입력**: 인시던트 ID, 에스컬레이션 대상·유형(계층적/기능적)
- **출력**: 이관 결과, 이력
- **처리 흐름**:
  1. 대상 배정·통지
  2. 이력 기록
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 에스컬레이션하면, 시스템은 대상에게 배정·통지하고 이력에 기록해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 대상 담당자가 존재하지 않으면, **THEN** 시스템은 400 오류를 반환해야 한다.

### FEAT-INC-006 · 상태 커뮤니케이션·타임라인

- **설명**: 상태 업데이트(내부/외부 구분)를 등록하고 타임라인에 기록한다.
- **관련 요구사항**: REQ-INC-006
- **입력**: 인시던트 ID, 업데이트 내용, 공개 범위(내부/외부)
- **출력**: 타임라인 항목
- **처리 흐름**:
  1. 업데이트 저장(타임스탬프)
  2. 타임라인에 추가
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 업데이트를 등록하면, 시스템은 타임스탬프와 함께 타임라인에 기록해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 접근 권한이 없는 사용자가 업데이트를 등록하면, **THEN** 시스템은 403을 반환해야 한다.

### FEAT-INC-007 · 해결·시간 지표

- **설명**: 해결 처리 후 시간 지표를 계산한다.
- **관련 요구사항**: REQ-INC-007
- **입력**: 인시던트 ID, 영향시작·탐지·영향종료 시각
- **출력**: 해결 상태, MTTD/MTTA/MTTR 값
- **처리 흐름**:
  1. 상태=해결 전이, 해결 시각 기록
  2. 시각 데이터로 지표 계산
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 인시던트가 해결되면, 시스템은 해결 시각 기록 및 지표를 계산해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 지표 계산에 필요한 시각이 없으면, **THEN** 시스템은 해당 지표를 미산정으로 표시해야 한다.

### FEAT-INC-008 · 포스트모템

- **설명**: 인시던트의 포스트모템을 작성한다(blameless).
- **관련 요구사항**: REQ-INC-008
- **입력**: 인시던트 ID, 요약·타임라인·5 Whys·근본원인·조치항목
- **출력**: 저장된 포스트모템
- **처리 흐름**:
  1. 포스트모템 작성·인시던트 연결
  2. 근본원인·조치항목 저장
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 포스트모템을 제출하면, 시스템은 이를 인시던트에 연결·저장해야 한다.
  - (State-driven) **WHILE** SEV 1·2가 해결됐으나 포스트모템이 없는 동안, 시스템은 "포스트모템 필요"로 표시해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 필수 항목(근본원인)이 비어 제출되면, **THEN** 시스템은 제출을 거부해야 한다.

### FEAT-INC-009 · 문제 연계

- **설명**: 인시던트를 문제(Problem)로 연결·등록한다.
- **관련 요구사항**: REQ-INC-009
- **입력**: 인시던트 ID, (신규 또는 기존)문제 ID
- **출력**: 인시던트-문제 링크
- **처리 흐름**:
  1. 신규 문제 생성 또는 기존 문제 선택
  2. 링크 저장
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 문제 등록/연결을 실행하면, 시스템은 인시던트와 문제를 링크해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 존재하지 않는 문제에 연결하면, **THEN** 시스템은 400 오류를 반환해야 한다.

### FEAT-INC-010 · 지표 리포팅

- **설명**: 인시던트 지표를 집계·조회한다.
- **관련 요구사항**: REQ-INC-010
- **입력**: 조회 기간·필터
- **출력**: 건수·심각도 분포·평균 MTTR
- **처리 흐름**:
  1. 데이터 집계
  2. 지표 계산·반환
- **인수 기준 (EARS)**:
  - (Ubiquitous) 시스템은 기간별 건수·심각도 분포·평균 MTTR를 조회할 수 있어야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 데이터가 없으면, **THEN** 시스템은 빈 결과를 반환해야 한다.
