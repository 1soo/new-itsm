# 기능 명세서 — 서비스 요청 관리 (Service Request Management)

> 도메인: service-request · 버전: 0.1 · 작성일: 2026-07-09

## 1. 개요

서비스 카탈로그·셀프서비스 포털·요청 7단계 워크플로우·큐 배정·승인·SLA·CSAT·지식 연계 기능을 제공하여 일상적 서비스 요청을 표준화·자동화한다.

## 3. 기능 상세

### FEAT-SRM-001 · 서비스 카탈로그 관리

- **설명**: 프로세스 오너가 요청 유형(카탈로그 항목)을 정의·정리한다.
- **관련 요구사항**: REQ-SRM-001
- **입력**: 요청 유형명·설명·양식 필드 정의·승인 필요 여부·담당 큐·SLA 목표
- **출력**: 등록/수정된 요청 유형, 카탈로그 목록
- **처리 흐름**:
  1. 프로세스 오너 권한 검증
  2. 요청 유형 정보·양식 스키마 저장
  3. 카탈로그(그룹)에 배치
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 유효한 요청 유형을 생성하면, 시스템은 카탈로그에 등록해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 필수 항목(이름·양식)이 누락되면, **THEN** 시스템은 400 오류를 반환해야 한다.
  - **IF** 프로세스 오너 권한이 없으면, **THEN** 시스템은 403을 반환해야 한다.

### FEAT-SRM-002 · 요청 제출

- **설명**: 요청자가 포털에서 요청 유형을 선택하고 양식을 작성해 제출한다.
- **관련 요구사항**: REQ-SRM-002
- **입력**: 요청 유형, 양식 필드 값
- **출력**: 생성된 요청(접수번호·초기 상태)
- **처리 흐름**:
  1. 요청 유형 양식 로드
  2. 입력값 유효성 검사
  3. 요청 생성(상태=제출됨)
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 양식을 완성해 제출하면, 시스템은 요청을 생성하고 접수번호를 반환해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 필수 필드가 미입력이면, **THEN** 시스템은 제출을 거부해야 한다.

### FEAT-SRM-003 · 요청 검증

- **설명**: 제출된 요청의 완전성·자격을 검증한다.
- **관련 요구사항**: REQ-SRM-003
- **입력**: 제출된 요청
- **출력**: 검증 통과(다음 단계) 또는 반려
- **처리 흐름**:
  1. 필수 정보 완전성 확인
  2. 요청자 자격 확인
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 요청이 제출되면, 시스템은 완전성·자격을 검증해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 자격 미달이면, **THEN** 시스템은 요청을 반려하고 사유를 표시해야 한다.

### FEAT-SRM-004 · 분류·라우팅

- **설명**: 검증 통과 요청을 규칙에 따라 큐/팀에 배정한다.
- **관련 요구사항**: REQ-SRM-004
- **입력**: 검증 통과 요청, 요청 유형-큐 매핑
- **출력**: 큐에 배정된 요청
- **처리 흐름**:
  1. 요청 유형 기반 큐 결정
  2. 해당 큐로 라우팅
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 요청이 검증을 통과하면, 시스템은 적절한 큐로 라우팅해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 매칭 큐가 없으면, **THEN** 시스템은 기본(미분류) 큐로 배정해야 한다.

### FEAT-SRM-005 · 승인 처리

- **설명**: 승인이 필요한 요청 유형은 승인자에게 승인 요청을 보낸다.
- **관련 요구사항**: REQ-SRM-005
- **입력**: 승인 대상 요청, 승인자, 승인/반려 결정·사유
- **출력**: 승인/반려 결과, 갱신된 상태
- **처리 흐름**:
  1. 요청 유형의 승인 필요 여부 확인
  2. 승인자에게 승인 대기 알림
  3. 승인/반려 처리
- **인수 기준 (EARS)**:
  - (State-driven) **WHILE** 승인 대기 상태인 동안, 시스템은 이행으로 진행하지 않아야 한다.
  - (Event-driven) **WHEN** 승인자가 결정하면, 시스템은 상태를 갱신해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 반려되면, **THEN** 시스템은 요청을 반려 처리하고 요청자에게 통지해야 한다.
  - **IF** 지정 승인자가 아닌 사용자가 승인하려 하면, **THEN** 시스템은 403을 반환해야 한다.

### FEAT-SRM-006 · 이행·배정

- **설명**: 상담원이 큐에서 요청을 배정받아 이행하고 상태를 갱신한다.
- **관련 요구사항**: REQ-SRM-006
- **입력**: 요청 ID, 담당자, 처리 내용
- **출력**: 갱신된 요청 상태
- **처리 흐름**:
  1. 상담원 배정
  2. 이행 작업 수행·기록
  3. 상태=이행 완료로 갱신
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 상담원이 이행을 완료하면, 시스템은 상태를 갱신해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 권한 없는 사용자가 배정/이행하려 하면, **THEN** 시스템은 403을 반환해야 한다.

### FEAT-SRM-007 · 검증·종료

- **설명**: 완료를 확인하고 요청을 종료한다.
- **관련 요구사항**: REQ-SRM-007
- **입력**: 요청 ID, 완료 확인
- **출력**: 종료된 요청
- **처리 흐름**:
  1. 완료 여부 확인
  2. 상태=종료로 전환·기록
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 완료가 확인되면, 시스템은 요청을 종료해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 이미 종료된 요청을 종료하려 하면, **THEN** 시스템은 중복 처리를 거부해야 한다.

### FEAT-SRM-008 · SLA 관리

- **설명**: 요청 유형별 SLA 목표를 기준으로 경과 시간을 추적하고 위반을 표시한다.
- **관련 요구사항**: REQ-SRM-008
- **입력**: 요청, SLA 목표(응답·해결)
- **출력**: SLA 상태(준수/임박/위반)
- **처리 흐름**:
  1. 요청 생성 시 SLA 타이머 시작
  2. 경과 시간 계산·목표 비교
- **인수 기준 (EARS)**:
  - (Ubiquitous) 시스템은 각 요청의 SLA 경과 시간을 계산해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** SLA 목표 시간을 초과하면, **THEN** 시스템은 SLA 위반으로 표시해야 한다.

### FEAT-SRM-009 · 요청 코멘트

- **설명**: 요청자와 상담원이 요청에서 코멘트로 소통한다.
- **관련 요구사항**: REQ-SRM-009
- **입력**: 요청 ID, 코멘트 내용
- **출력**: 요청 이력에 추가된 코멘트
- **처리 흐름**:
  1. 코멘트 저장
  2. 상대에게 노출
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 코멘트가 등록되면, 시스템은 이력에 추가하고 상대에게 표시해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 요청 접근 권한이 없는 사용자가 코멘트를 달면, **THEN** 시스템은 403을 반환해야 한다.

### FEAT-SRM-010 · CSAT 수집

- **설명**: 요청 종료 후 요청자에게 만족도 평가를 요청한다.
- **관련 요구사항**: REQ-SRM-010
- **입력**: 종료된 요청, CSAT 점수
- **출력**: 저장된 CSAT
- **처리 흐름**:
  1. 종료 시 CSAT 요청 생성
  2. 응답 저장·집계
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 요청이 종료되면, 시스템은 CSAT 평가를 요청해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 종료되지 않은 요청에 CSAT를 제출하면, **THEN** 시스템은 이를 거부해야 한다.

### FEAT-SRM-011 · 지식 기사 추천(티켓 차단)

- **설명**: 요청 생성 과정에서 관련 지식 기사를 추천해 티켓을 차단한다.
- **관련 요구사항**: REQ-SRM-011
- **입력**: 요청 유형/제목/키워드
- **출력**: 추천 지식 기사 목록
- **처리 흐름**:
  1. 입력 기반 지식 검색
  2. 관련 기사 노출
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 요청자가 유형 선택·제목 입력을 하면, 시스템은 관련 기사를 추천해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 매칭 기사가 없으면, **THEN** 시스템은 추천 없이 요청 제출을 계속 진행할 수 있어야 한다.

### FEAT-SRM-012 · 지표 리포팅

- **설명**: CSAT·응답/해결시간·SLA 준수율을 집계·조회한다.
- **관련 요구사항**: REQ-SRM-012
- **입력**: 조회 기간·필터
- **출력**: 지표 집계 결과
- **처리 흐름**:
  1. 요청 데이터 집계
  2. 지표 계산·반환
- **인수 기준 (EARS)**:
  - (Ubiquitous) 시스템은 CSAT·평균 응답/해결시간·SLA 준수율을 집계·조회할 수 있어야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 조회 기간에 데이터가 없으면, **THEN** 시스템은 빈 결과(0값)를 반환해야 한다.
