# 기능 명세서 — 변경 관리 (Change Management / Enablement)

## 3. 기능 상세

### FEAT-CHG-001 · 변경 요청(RFC) 생성

- **설명**: 요청자가 RFC를 생성하고 위험·구현·영향·롤백을 기입한다.
- **관련 요구사항**: REQ-CHG-001
- **입력**: 요약, 설명, 변경 유형, 위험, 예상 구현, 영향 시스템, 롤백 방법
- **출력**: 생성된 변경(식별키·초기 상태 요청)
- **처리 흐름**:
  1. 입력값 검증
  2. 변경 생성
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 필수 정보로 RFC를 생성하면, 시스템은 변경을 생성해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 요약·변경 유형이 누락되면, **THEN** 시스템은 400 오류를 반환해야 한다.

### FEAT-CHG-002 · 변경 유형 분류

- **설명**: 변경을 표준·일반·긴급으로 분류한다.
- **관련 요구사항**: REQ-CHG-002
- **입력**: 변경 ID, 변경 유형
- **출력**: 분류된 변경
- **처리 흐름**:
  1. 유형 지정
  2. 승인 경로 연동
- **인수 기준 (EARS)**:
  - (Ubiquitous) 시스템은 변경을 세 유형 중 하나로 분류·표시해야 한다.
  - (Event-driven) **WHEN** 유형이 지정되면, 시스템은 승인 경로를 연동해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 정의되지 않은 유형을 지정하면, **THEN** 시스템은 거부해야 한다.

### FEAT-CHG-003 · 변경 프로세스 상태 관리

- **설명**: 변경을 6단계 프로세스에 따라 전이한다.
- **관련 요구사항**: REQ-CHG-003
- **입력**: 변경 ID, 목표 단계
- **출력**: 갱신된 상태
- **처리 흐름**:
  1. 전이 허용 여부 확인(승인 완료 여부 포함)
  2. 전이·이력 기록
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 허용된 전이를 요청하면, 시스템은 상태를 갱신해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 승인 완료 전 구현으로 전이하려 하면, **THEN** 시스템은 거부해야 한다.

### FEAT-CHG-004 · 승인 경로 결정

- **설명**: 변경 유형·위험도에 따라 승인 경로를 결정한다.
- **관련 요구사항**: REQ-CHG-004
- **입력**: 변경 유형, 위험도
- **출력**: 결정된 승인 경로(자동/동료검토/CAB)
- **처리 흐름**:
  1. 유형·위험 평가
  2. 승인 경로 지정
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 검토 단계에 진입하면, 시스템은 승인 경로를 결정해야 한다.
  - (Complex) **WHILE** 고위험인 동안, **WHEN** 경로가 결정되면, 시스템은 CAB 승인을 요구해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 위험도가 미평가면, **THEN** 시스템은 기본(CAB) 경로로 처리해야 한다.

### FEAT-CHG-005 · CAB 승인 처리

- **설명**: CAB/승인자가 변경을 검토·승인·반려한다.
- **관련 요구사항**: REQ-CHG-005
- **입력**: 변경 ID, 승인/반려 결정·의견
- **출력**: 갱신된 승인 상태·이력
- **처리 흐름**:
  1. 승인 권한 확인
  2. 결정 반영·이력 기록
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 승인 권한자가 결정하면, 시스템은 상태를 갱신하고 이력을 기록해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 승인 권한이 없으면, **THEN** 시스템은 403을 반환해야 한다.

### FEAT-CHG-006 · 표준 변경 사전승인

- **설명**: 사전 승인된 표준 변경은 재승인 없이 진행한다.
- **관련 요구사항**: REQ-CHG-006
- **입력**: 표준 변경(사전승인 템플릿)
- **출력**: 승인 단계 생략된 변경
- **처리 흐름**:
  1. 표준 변경 여부 확인
  2. 승인 단계 자동 통과
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 사전 승인 표준 변경이 생성되면, 시스템은 승인 없이 계획/구현으로 진행하게 해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 표준 템플릿에 없는 변경을 표준으로 지정하면, **THEN** 시스템은 일반 변경 승인 경로로 처리해야 한다.

### FEAT-CHG-007 · 변경 일정(캘린더)

- **설명**: 예정 변경을 캘린더로 조회한다.
- **관련 요구사항**: REQ-CHG-007
- **입력**: 조회 기간
- **출력**: 기간 내 예정 변경 목록
- **처리 흐름**:
  1. 기간 내 변경 조회
  2. 일정 표시
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 변경 일정을 조회하면, 시스템은 예정 변경을 기간별로 표시해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 예정 변경이 없으면, **THEN** 시스템은 빈 일정을 반환해야 한다.

### FEAT-CHG-008 · 구현 결과 기록

- **설명**: 변경 구현 결과와 롤백 여부를 기록한다.
- **관련 요구사항**: REQ-CHG-008
- **입력**: 변경 ID, 결과(성공/실패), 롤백 여부, 비고
- **출력**: 저장된 구현 결과
- **처리 흐름**:
  1. 구현 결과 입력
  2. 변경에 저장·종료 준비
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 결과·롤백 여부를 기록하면, 시스템은 결과를 저장해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 승인되지 않은 변경에 결과를 기록하려 하면, **THEN** 시스템은 거부해야 한다.

### FEAT-CHG-009 · 인시던트/문제 연계

- **설명**: 변경을 인시던트/문제에 연결한다.
- **관련 요구사항**: REQ-CHG-009
- **입력**: 변경 ID, 인시던트/문제 ID
- **출력**: 링크
- **처리 흐름**:
  1. 대상 확인
  2. 링크 저장
- **인수 기준 (EARS)**:
  - (Event-driven) **WHEN** 변경을 인시던트/문제에 연결하면, 시스템은 링크를 저장해야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 존재하지 않는 대상에 연결하면, **THEN** 시스템은 400 오류를 반환해야 한다.

### FEAT-CHG-010 · 지표 리포팅

- **설명**: 변경 성공률·실패율·긴급 변경 비율을 집계·조회한다.
- **관련 요구사항**: REQ-CHG-010
- **입력**: 조회 기간·필터
- **출력**: 지표 집계 결과
- **처리 흐름**:
  1. 변경 데이터 집계
  2. 지표 계산·반환
- **인수 기준 (EARS)**:
  - (Ubiquitous) 시스템은 변경 성공률·실패율·긴급 변경 비율을 조회할 수 있어야 한다.
- **예외/오류 처리 (EARS · Unwanted Behaviour)**:
  - **IF** 데이터가 없으면, **THEN** 시스템은 빈 결과를 반환해야 한다.
