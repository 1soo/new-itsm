## 1. 개요

셀프서비스 포털에서의 요청 제출(동적 양식·지식 추천), 요청 큐/상세(검증·라우팅·커스텀 승인·이행·검증·종료), SLA·코멘트·CSAT, 서비스 카탈로그 관리(카탈로그 카테고리 고정 목록 CRUD 포함), 지표 리포팅 화면을 정의한다. 승인 대기·결정은 [common.md](common.md) SCR-COM-014(전 도메인 공용)를 사용한다.

## 4. 화면 상세

### SCR-SRM-001 · 서비스 포털(카탈로그 브라우즈)

- **목적**: 요청자가 카탈로그에서 요청 유형을 탐색·선택.
- **레이아웃**: 상단 검색바 / 카테고리 그룹 카드 그리드 / 각 카드에 요청 유형명·설명.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 카탈로그 카드 | 카드 | 요청 유형 선택 진입 | Base(강조) |
  | 검색바 | 입력 | 유형/키워드 검색 | Border/Base |
- **상태 · 인터랙션**: 카드 클릭 시 SCR-SRM-002 이동. 검색 시 관련 지식 기사 추천 배너 노출(있을 때만).
- **연관 API**: `GET /api/v1/service-catalog/items`

### SCR-SRM-002 · 요청 제출(동적 양식)

- **목적**: 선택한 요청 유형의 동적 양식을 작성해 제출.
- **레이아웃**: 좌측 동적 폼(요청 유형 스키마 기반) / 우측 "관련 지식 기사" 추천 패널 / 하단 제출 버튼.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 동적 필드 | 입력/셀렉트/첨부 | 유형별 양식 스키마 렌더 | Border/Base |
  | 지식 추천 패널 | 리스트 | 제목/유형 기반 기사 추천(deflection) | Variation 2 |
  | 제출 버튼 | 버튼 | 요청 생성 | Base |
- **상태 · 인터랙션**: 필수 필드 미입력 시 제출 차단·인라인 오류. 추천 기사 없으면 패널 숨김. 제출 성공 시 접수번호 토스트·상세 이동.
- **연관 API**: `GET /api/v1/service-catalog/items/{id}`, `GET /api/v1/knowledge/suggestions`, `POST /api/v1/service-requests`

### SCR-SRM-003 · 내 요청 목록

- **목적**: 요청자가 본인 요청 상태를 추적.
- **레이아웃**: [SCR-COM-007] 목록 패턴. 필터(상태·기간) / 표(접수번호·유형·상태·SLA·갱신일).
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 상태 배지 | 배지 | 제출/승인대기/이행중/종료 | 상태 시맨틱 |
  | SLA 배지 | 배지 | 준수/임박/위반 | Success/Warning/Danger |
- **상태 · 인터랙션**: 본인 요청만 조회. 행 클릭 시 상세.
- **컬럼 폭(유지보수 요청, 2026-07-14, [common.md](common.md) SCR-COM-007 아키텍처 적용)**:
  | 컬럼 | 폭(px) | 근거 |
  |------|--------|------|
  | 접수번호 | 130 | 코드(ticketKey) |
  | 유형 | 미지정(auto) | 카탈로그 항목명 길이 가변, 잔여 폭 흡수. `truncate` 적용 |
  | 상태 | 110 | 상태 뱃지 |
  | SLA | 110 | SLA 뱃지 |
  | 담당자 | 120 | 사람 이름/"미배정" |
  | 갱신일 | 110 | 날짜(YYYY-MM-DD) |
- **페이지당 아이템 수(유지보수 요청, 2026-07-14)**: 10건 → 13건([common.md](common.md) 7절 패턴 A).
- **연관 API**: `GET /api/v1/service-requests?scope=mine`

### SCR-SRM-004 · 요청 큐(상담원)

- **목적**: 상담원이 큐별 요청을 배정받아 처리.
- **레이아웃**: 좌측 큐 목록 / 우측 [SCR-COM-007] 요청 표. 미분류 큐 표시.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 큐 목록 | 리스트 | 큐별 건수 | Neutral |
  | 배정 버튼 | 버튼 | 클릭 시 담당자 배정 팝업(아래) 오픈. 라벨은 "배정"(기존 "나에게 배정"에서 변경, 2026-07-15 유지보수 요청 — 역할 기반 후보 선택도 겸하므로) | Base |
  | 담당자 배정 팝업 | 모달([공용 Modal](common.md) 재사용) | 대상 요청의 카탈로그 항목에 담당자 역할이 지정되어 있으면 역할 보유자 후보 목록(API-SRM-017, 이름 클릭)을 보여준다. 후보가 없거나(역할 미지정 카탈로그 항목) 목록에 본인이 있으면 "나에게 배정" 버튼도 함께 제공한다. 선택 확정 시 API-SRM-009(assigneeId) 호출 | Base |
  | 미분류 표시 | 배지 | 매칭 큐 없음 | Warning |
- **상태 · 인터랙션**:
  - 권한 없는 사용자는 배정 불가(403). 배정 시 담당자 갱신.
  - **배정 버튼 노출 조건(2026-07-15 유지보수 요청, 신규)**: 아래 2가지 중 하나라도 해당하면 해당 행의 배정 버튼을 숨긴다 — (1) 이미 로그인 사용자 본인에게 배정됨(목록 응답의 `assigneeId`가 로그인 사용자 id와 일치), (2) 요청 상태가 라우팅(ROUTED) 이후(ROUTED/IN_FULFILLMENT/FULFILLED/CLOSED 중 하나).
- **컬럼 폭(유지보수 요청, 2026-07-14, [common.md](common.md) SCR-COM-007 아키텍처 적용)**:
  | 컬럼 | 폭(px) | 근거 |
  |------|--------|------|
  | 접수번호 | 130 | 코드(ticketKey) |
  | 유형 | 미지정(auto) | 카탈로그 항목명 길이 가변, 잔여 폭 흡수. `truncate` 적용 |
  | 상태 | 110 | 상태 뱃지 |
  | SLA | 110 | SLA 뱃지 |
  | 담당자 | 120 | 사람 이름/"미배정" |
  | 갱신일 | 110 | 날짜(YYYY-MM-DD) |
  | (액션) | 140 | "배정" 버튼 1개(노출 조건 충족 시만), 우측 정렬 |
- **페이지당 아이템 수(유지보수 요청, 2026-07-14)**: 10건 → 16건([common.md](common.md) 7절 패턴 D).
- **연관 API**: `GET /api/v1/service-requests?queue=`, `GET /api/v1/service-requests/{id}/assignee-candidates`(API-SRM-017), `POST /api/v1/service-requests/{id}/assign`

### SCR-SRM-005 · 요청 상세

- **목적**: 요청 처리 전 과정(검증·승인 상태·이행·검증/종료·코멘트·SLA·CSAT) 관리.
- **레이아웃**: [SCR-COM-008] 상세 패턴. 본문(양식 값)·코멘트·타임라인 / 우측 메타(상태·담당자·SLA 타이머·승인 패널(공용)·연결 지식).
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 상태 전이 버튼 | 버튼 | 이행완료/종료 등 허용 전이. 라우팅(ROUTED) 전이 버튼은 담당자 미배정 시 비활성화(`vulnerability` 도메인의 담당자 미배정 상태 REMEDIATION 전이 가드 패턴 재사용 — 비활성화+안내 tooltip). 라벨은 도착 상태명이 아닌 동작 동사형([common.md](common.md) SCR-COM-008 아키텍처 적용) — 아래 표 | Base |
  | 승인 패널(공용) | 패널 | 매칭되는 승인 프로세스가 있으면 차수 진행 상태(API-COM-004)·반려 사유 표시. 없으면 상태에 따라 문구 분기(2026-07-15 유지보수 요청, 아래 참조) | Info/Warning/Success/Danger |
  | SLA 타이머 | 배지 | 응답·해결 경과/위반 | Success/Warning/Danger |
  | 코멘트 입력 | 입력 | 요청자-상담원 소통 | Border/Base |
  | CSAT 위젯 | 별점/버튼 | 종료 시 요청자 노출 | Success |
- **상태 · 인터랙션**:
  - **승인 패널 미매칭 문구 분리(2026-07-15 유지보수 요청, UX 개선)**: 승인 게이트는 요청이 ROUTED→IN_FULFILLMENT 전이를 시도하는 시점에만 평가되므로(승인 인스턴스가 이때 처음 생성됨), 요청 상태가 아직 SUBMITTED/VALIDATED/ROUTED(게이트 평가 전)인 동안은 실제 매칭 규칙 유무와 무관하게 항상 인스턴스가 없어 "승인 절차가 없다"고 오해를 유발했다. 이제 요청 상태 기준으로 문구를 분리한다: 게이트 평가 전(SUBMITTED/VALIDATED/ROUTED)에는 중립 문구("이행 단계로 전환 시 승인 절차 적용 여부가 결정됩니다" 등, 미확정 안내)를, 게이트 평가 후(IN_FULFILLMENT 이상)에 인스턴스가 없으면 기존 문구("이 요청에는 승인 절차가 없습니다")를 표시한다. 승인 패널 컴포넌트 자체(prop 구조)는 변경하지 않고, 이 화면이 요청 상태에 따라 주입하는 문구 텍스트만 분기한다.
  - 승인 대기 중에는 이행 전이 버튼 숨김(매칭되는 프로세스가 없으면 게이트 없이 진행). 이미 종료된 요청 재종료 차단. CSAT는 종료 후 요청자에게만 노출. 권한 없는 코멘트/이행 시 403. 승인 대기 중인 건의 처리는 [common.md](common.md) SCR-COM-014에서 수행(이 화면은 진행 상태 조회 전용).
  - **라우팅(ROUTED) 버튼 비활성화(2026-07-15 유지보수 요청)**: 담당자(`assignee`)가 배정되지 않은 상태에서는 라우팅 전이 버튼을 비활성화하고 안내 tooltip("담당자 미배정 상태로는 라우팅 단계로 전이할 수 없습니다")을 표시한다. 배정은 SCR-SRM-004에서 수행한다. 서버도 동일 조건을 검증해 409를 반환한다(API-SRM-010 참조).
  - **전이 버튼 라벨**: `status.ts`의 `transitionLabel(t, target)`(i18n 키 `service-request:transition.*`)이 버튼 텍스트를 결정한다(전이 완료 토스트 문구는 `statusLabel(t, target)`을 사용, [common.md](common.md) SCR-COM-008 아키텍처 참조).
    | 도착 상태 | 버튼 라벨 |
    |-----------|-----------|
    | VALIDATED | 검증 완료 |
    | ROUTED | 라우팅 처리 |
    | IN_FULFILLMENT | 이행 시작 |
    | FULFILLED | 이행 완료 처리 |
    | CLOSED | 종료 처리 |
  - **타임라인 actor 표시**: 타임라인 각 항목에 행위 수행 주체자를 함께 표시한다([common.md](common.md) SCR-COM-008 아키텍처 참조, `RequestDetailResponse.TimelineEntry`의 `actor` 필드). 상태 변경 타임라인 메시지는 `target.name()`(코드) 대신 상태 라벨을 사용한다.
- **연관 API**: `GET /api/v1/service-requests/{id}`, `PATCH .../status`, `POST .../comments`, `POST .../csat`, `GET /api/v1/approvals/{approvalRequestId}`(API-COM-004)

### SCR-SRM-007 · 서비스 카탈로그 관리

- **목적**: 프로세스 오너가 요청 유형(양식·SLA·큐·담당자 역할·카테고리)을 정의·정리. 승인 여부·경로 설정은 이 화면이 아닌 SCR-ADMIN-008(승인 프로세스 생성/편집, SYSTEM_ADMIN 전용)에서 도메인=SERVICE_REQUEST·요청유형=이 카탈로그 항목으로 별도 구성한다(승인 프로세스 커스텀 기능으로 완전 대체).
- **레이아웃**: 상단 탭("카탈로그 항목"/"카테고리 관리" — 카테고리 관리는 SCR-SRM-009로 별도 기술). "카탈로그 항목" 탭: 카탈로그 항목 목록 / 항목 편집 폼(이름·설명·카테고리·양식 필드 빌더·담당 큐·담당자 역할·SLA 목표).
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 카테고리 선택 | Select | 카테고리 목록(API-SRM-018, SCR-SRM-009에서 관리) 중 선택, 미선택="미분류" | Border/Base |
  | 담당 큐 선택 | Select | 큐 목록(API-SRM-016) 중 선택, 미선택="미분류" | Border/Base |
  | 담당자 역할 선택 | Select | 역할 목록(API-AUTH-030) 중 선택(선택 항목). 지정 시 요청 큐(SCR-SRM-004)의 담당자 배정 팝업이 이 역할 보유자를 후보로 노출(자동배정 아님) | Border/Base |
  | 양식 필드 빌더 | 반복 입력 | 동적 필드 정의(라벨·유형·필수). 필드 유형에 "여러 줄 텍스트"(textarea) 포함(`components/common/field-builder.tsx`·`dynamic-form.tsx`·`form-schema.ts`의 `FormFieldType` 공용 계약 — SRM+ESM 동적 폼 모두 적용) | Border/Base |
  | SLA 목표 입력 | 입력 | 응답·해결 시간 | Border/Base |
- **상태 · 인터랙션**: 이름·양식 누락 시 400. 프로세스 오너 권한 없으면 403. 항목 편집 진입 시 카테고리·담당 큐·담당자 역할 select는 상세 조회 응답(`categoryId`/`queueId`/`assigneeRoleId`)의 기존 값으로 프리필한다.
- **연관 API**: `GET/POST/PATCH /api/v1/service-catalog/items`, `GET /api/v1/queues`(API-SRM-016, 큐 select 후보), `GET /api/v1/roles`(API-AUTH-030, 담당자 역할 select 후보), `GET /api/v1/service-catalog/categories`(API-SRM-018, 카테고리 select 후보)

### SCR-SRM-009 · 서비스 카탈로그 카테고리 관리

- **목적**: 프로세스 오너가 카탈로그 항목에서 선택 가능한 카테고리 고정 목록을 관리한다.
- **레이아웃**: SCR-SRM-007과 같은 화면의 "카테고리 관리" 탭. 목록(이름·표시 순서·사용 중인 카탈로그 항목 수) / 상단 "새 카테고리" 버튼 / 행별 인라인 수정·삭제 버튼.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 카테고리 목록 표 | 표 | 이름·표시 순서·사용 항목 수 | `--foreground`/`--card` |
  | 새 카테고리 버튼 | 버튼 | 이름·표시 순서 입력 모달([공용 Modal](common.md) 재사용) 오픈 | Base |
  | 수정 버튼 | 버튼(행별) | 동일 모달을 값 프리필로 오픈 | Border/Base |
  | 삭제 버튼 | 버튼(행별) | 확인 다이얼로그(SCR-COM-009) 후 삭제 시도 | Danger |
- **상태 · 인터랙션**: 이름 누락·중복 시 400/409. 프로세스 오너 권한 없으면 403. 하나 이상의 카탈로그 항목이 참조 중인 카테고리 삭제 시도 시 409(`CATEGORY_IN_USE`) — 오류 토스트로 "이 카테고리를 사용하는 카탈로그 항목이 있어 삭제할 수 없습니다. 항목의 카테고리를 먼저 변경하세요" 안내(참조 중인 항목을 자동으로 미분류 처리하지 않음, 관리자가 명시적으로 재배정하도록 강제). 삭제·수정 성공 시 카탈로그 항목 편집 폼(SCR-SRM-007)의 카테고리 select 후보 목록도 즉시 갱신되어야 하므로 두 탭이 같은 카테고리 목록 조회 API를 공유한다.
- **연관 API**: `GET /api/v1/service-catalog/categories`(API-SRM-018), `POST /api/v1/service-catalog/categories`(API-SRM-019), `PATCH /api/v1/service-catalog/categories/{id}`(API-SRM-020), `DELETE /api/v1/service-catalog/categories/{id}`(API-SRM-021)
