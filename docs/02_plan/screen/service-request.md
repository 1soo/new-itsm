## 1. 개요

셀프서비스 포털에서의 요청 제출(동적 양식·지식 추천), 요청 처리함/상세(카테고리별 배정, 검증·라우팅·커스텀 승인·이행·검증·종료), SLA·코멘트·CSAT, 서비스 카탈로그 관리(카탈로그 카테고리 고정 목록 CRUD 포함), 지표 리포팅 화면을 정의한다. 승인 대기·결정은 [common.md](common.md) SCR-COM-014(전 도메인 공용)를 사용한다.

## 4. 화면 상세

### SCR-SRM-001 · 서비스 포털(카탈로그 브라우즈)

- **목적**: 요청자가 카탈로그에서 요청 유형을 탐색·선택.
- **레이아웃**: 상단 검색바 / 카테고리 그룹 카드 그리드 / 각 카드에 요청 유형명·설명.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 카탈로그 카드 | 카드 | 요청 유형 선택 진입 | Base(강조) |
  | 검색바 | 입력 | 유형/키워드 검색 | Border/Base |
- **상태 · 인터랙션**: 카드 클릭 시 SCR-SRM-002 이동. 검색 시 관련 지식 기사 추천 배너 노출(있을 때만).
- **연관 API**: `GET /api/v1/service-catalog/items`

### SCR-SRM-002 · 요청 제출(동적 양식)

- **목적**: 선택한 요청 유형의 동적 양식을 작성해 제출.
- **레이아웃**: 좌측 동적 폼(카탈로그 항목의 `form_schema` 그리드 배치 그대로 렌더) / 우측 "관련 지식 기사" 추천 패널 / 하단 제출 버튼.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 동적 폼 렌더러 | `dynamic-form-renderer.tsx`(5절 공용, 자체 8×n 그리드 엔진) | `formSchema`(API-SRM-002)의 그리드 배치·Content 설정을 그대로 렌더(date/file은 아이콘+선택값 텍스트로 렌더, 5.4절). 클라이언트 검증(필수·정규식)은 실시간이 아니라 제출 클릭 시 순차 1건 표시(5.5절) | Border/Base |
  | 유효성 오류 메시지 | 텍스트(폼 하단 공통 위치) | 제출 클릭 시 `components` 배열 순서상 첫 번째 위반 컴포넌트의 오류 1건만 표시(필드별 인라인 오류 없음) | Danger |
  | 지식 추천 패널 | 리스트 | 제목/유형 기반 기사 추천(deflection) | Variation 2 |
  | 제출 버튼 | 버튼 | 렌더러의 제출 데이터 수신 후 요청 생성 호출 | Base |
- **상태 · 인터랙션**: 형식·필수 위반 시 제출을 막고 폼 하단에 첫 번째 위반 오류 1건만 표시한다(5.5절). 재제출 시 처음부터 다시 검사해 다음 위반을 표시한다. 추천 기사 없으면 패널 숨김. 제출 성공 시 접수번호 토스트·상세 이동. 서버 재검증 실패(400)도 동일하게 첫 위반 1건 메시지를 공통 오류 토스트로 안내.
- **연관 API**: `GET /api/v1/service-catalog/items/{id}`, `GET /api/v1/knowledge/suggestions`, `POST /api/v1/service-requests`

### SCR-SRM-003 · 내 요청 목록

- **목적**: 요청자가 본인 요청 상태를 추적.
- **레이아웃**: [SCR-COM-007] 목록 패턴. 필터(상태·기간) / 표(접수번호·유형·상태·SLA·갱신일).
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 상태 배지 | 배지 | 제출/승인대기/이행중/종료 | 상태 시맨틱 |
  | SLA 배지 | 배지 | 준수/임박/위반 | Success/Warning/Danger |
- **상태 · 인터랙션**: 본인 요청만 조회. 행 클릭 시 상세.
- **컬럼 폭([common.md](common.md) SCR-COM-007 아키텍처 적용)**:
  | 컬럼 | 폭(px) | 근거 |
  |------|--------|------|
  | 접수번호 | 130 | 코드(ticketKey) |
  | 유형 | 미지정(auto) | 카탈로그 항목명 길이 가변, 잔여 폭 흡수. `truncate` 적용 |
  | 상태 | 110 | 상태 뱃지 |
  | SLA | 110 | SLA 뱃지 |
  | 담당자 | 120 | 사람 이름/"미배정" |
  | 갱신일 | 110 | 날짜(YYYY-MM-DD) |
- **페이지당 아이템 수**: 13건([common.md](common.md) 7절 패턴 A).
- **연관 API**: `GET /api/v1/service-requests?scope=mine`

### SCR-SRM-004 · 요청 처리함(상담원)

- **목적**: 상담원이 카테고리별로 요청을 배정받아 처리.
- **레이아웃**: 좌측 카테고리 목록(카탈로그 카테고리 고정 목록 + 미분류) / 우측 [SCR-COM-007] 요청 표.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 카테고리 목록 | 리스트 | 카테고리별 미종료 요청 건수(API-SRM-016), `sort_order` 오름차순 + 미분류 항목을 마지막에 고정 노출 | Neutral |
  | 배정 버튼 | 버튼 | 클릭 시 담당자 배정 팝업(아래) 오픈. 라벨은 "배정"(역할 기반 후보 선택도 겸함) | Base |
  | 담당자 배정 팝업 | 모달([공용 Modal](common.md) 재사용) | 대상 요청의 카탈로그 항목에 담당자 역할이 지정되어 있으면 역할 보유자 후보 목록(API-SRM-017, 이름 클릭)을 보여준다. 후보가 없거나(역할 미지정 카탈로그 항목) 목록에 본인이 있으면 "나에게 배정" 버튼도 함께 제공한다. 선택 확정 시 API-SRM-009(assigneeId) 호출 | Base |
  | 미분류 표시 | 배지 | 카탈로그 항목에 카테고리 미지정(`categoryId` null) | Warning |
- **상태 · 인터랙션**:
  - 권한 없는 사용자는 배정 불가(403). 배정 시 담당자 갱신.
  - 좌측 카테고리 목록 클릭 시 우측 표를 해당 카테고리(또는 미분류)로 서버 필터링(API-SRM-007 `categoryId=`). 요청의 카테고리는 스냅샷이 아니라 `catalogItemId → service_catalog_item.category_id` 실시간 조인 결과이므로, 카탈로그 항목의 카테고리를 변경하면 기존 요청의 분류도 즉시 반영된다.
  - **배정 버튼 노출 조건**: 아래 2가지 중 하나라도 해당하면 해당 행의 배정 버튼을 숨긴다 — (1) 이미 로그인 사용자 본인에게 배정됨(목록 응답의 `assigneeId`가 로그인 사용자 id와 일치), (2) 요청 상태가 라우팅(ROUTED) 이후(ROUTED/IN_FULFILLMENT/FULFILLED/CLOSED 중 하나).
- **컬럼 폭([common.md](common.md) SCR-COM-007 아키텍처 적용)**:
  | 컬럼 | 폭(px) | 근거 |
  |------|--------|------|
  | 접수번호 | 130 | 코드(ticketKey) |
  | 유형 | 미지정(auto) | 카탈로그 항목명 길이 가변, 잔여 폭 흡수. `truncate` 적용 |
  | 상태 | 110 | 상태 뱃지 |
  | SLA | 110 | SLA 뱃지 |
  | 담당자 | 120 | 사람 이름/"미배정" |
  | 갱신일 | 110 | 날짜(YYYY-MM-DD) |
  | (액션) | 140 | "배정" 버튼 1개(노출 조건 충족 시만), 우측 정렬 |

### SCR-SRM-005 · 요청 상세

- **목적**: 요청 처리 전 과정(검증·승인 상태·이행·검증/종료·코멘트·SLA·CSAT) 관리.
- **레이아웃**: [SCR-COM-008] 상세 패턴. 본문(양식 값)·코멘트·타임라인 / 우측 메타(상태·담당자·SLA 타이머·승인 패널(공용)·연결 지식).
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 상태 전이 버튼 | 버튼 | 이행완료/종료 등 허용 전이. 라우팅(ROUTED) 전이 버튼은 담당자 미배정 시 비활성화(`vulnerability` 도메인의 담당자 미배정 상태 REMEDIATION 전이 가드 패턴 재사용 — 비활성화+안내 tooltip). 라벨은 도착 상태명이 아닌 동작 동사형([common.md](common.md) SCR-COM-008 아키텍처 적용) — 아래 표 | Base |
  | 승인 패널(공용) | 패널 | 매칭되는 승인 프로세스가 있으면 차수 진행 상태(API-COM-004)·반려 사유 표시. 없으면 상태에 따라 문구 분기(아래 참조) | Info/Warning/Success/Danger |
  | SLA 타이머 | 배지 | 응답·해결 경과/위반 | Success/Warning/Danger |
  | 코멘트 입력 | 입력 | 요청자-상담원 소통 | Border/Base |
  | CSAT 위젯 | 별점/버튼 | 종료 시 요청자 노출 | Success |
- **상태 · 인터랙션**:
  - **승인 패널 미매칭 문구 분리**: 승인 게이트는 요청이 ROUTED→IN_FULFILLMENT 전이를 시도하는 시점에만 평가되므로(승인 인스턴스가 이때 처음 생성됨), 요청 상태가 아직 SUBMITTED/VALIDATED/ROUTED(게이트 평가 전)인 동안은 실제 매칭 규칙 유무와 무관하게 항상 인스턴스가 없어 "승인 절차가 없다"고 오해를 유발한다. 요청 상태 기준으로 문구를 분리한다: 게이트 평가 전(SUBMITTED/VALIDATED/ROUTED)에는 중립 문구("이행 단계로 전환 시 승인 절차 적용 여부가 결정됩니다" 등, 미확정 안내)를, 게이트 평가 후(IN_FULFILLMENT 이상)에 인스턴스가 없으면 기존 문구("이 요청에는 승인 절차가 없습니다")를 표시한다. 승인 패널 컴포넌트 자체(prop 구조)는 변경하지 않고, 이 화면이 요청 상태에 따라 주입하는 문구 텍스트만 분기한다.
  - 승인 대기 중에는 이행 전이 버튼 숨김(매칭되는 프로세스가 없으면 게이트 없이 진행). 이미 종료된 요청 재종료 차단. CSAT는 종료 후 요청자에게만 노출. 권한 없는 코멘트/이행 시 403. 승인 대기 중인 건의 처리는 [common.md](common.md) SCR-COM-014에서 수행(이 화면은 진행 상태 조회 전용).
  - **라우팅(ROUTED) 버튼 비활성화**: 담당자(`assignee`)가 배정되지 않은 상태에서는 라우팅 전이 버튼을 비활성화하고 안내 tooltip("담당자 미배정 상태로는 라우팅 단계로 전이할 수 없습니다")을 표시한다. 배정은 SCR-SRM-004에서 수행한다. 서버도 동일 조건을 검증해 409를 반환한다(API-SRM-010 참조).
  - **전이 버튼 라벨**: `status.ts`의 `transitionLabel(t, target)`(i18n 키 `service-request:transition.*`)이 버튼 텍스트를 결정한다(전이 완료 토스트 문구는 `statusLabel(t, target)`을 사용, [common.md](common.md) SCR-COM-008 아키텍처 참조).
    | 도착 상태 | 버튼 라벨 |
    |-----------|-----------|
    | VALIDATED | 검증 완료 |
    | ROUTED | 라우팅 처리 |
    | IN_FULFILLMENT | 이행 시작 |
    | FULFILLED | 이행 완료 처리 |
    | CLOSED | 종료 처리 |
  - **타임라인 actor 표시**: 타임라인 각 항목에 행위 수행 주체자를 함께 표시한다([common.md](common.md) SCR-COM-008 아키텍처 참조, `RequestDetailResponse.TimelineEntry`의 `actor` 필드). 상태 변경 타임라인 메시지는 `target.name()`(코드) 대신 상태 라벨을 사용한다.
- **연관 API**: `GET /api/v1/service-requests/{id}`, `PATCH .../status`, `POST .../comments`, `POST .../csat`, `GET /api/v1/approvals/{approvalRequestId}`(API-COM-004)

## 5. 동적 폼 빌더 아키텍처 (8×n 그리드, SRM 전용)

`@formio/js`/`@formio/react`(form.io)를 완전히 제거하고 자체 그리드 기반 드래그앤드롭 폼 빌더로 전면 재구현한다. 기존에는 이 아키텍처를 [common.md](common.md) 8절에 "SRM/ESM 공용"으로 정의했으나, ESM은 이 아키텍처가 적용된 적이 없고 기존 레거시 EAV(`esm_catalog_form_field`/`esm_request_form_value`)를 그대로 사용 중임이 확인되어(코드 조사) 이번에 **SRM 전용**으로 이 문서로 이전한다. 대상 화면: SCR-SRM-007("Form 설정" 팝업)·SCR-SRM-002(그리드 렌더러).

### 5.1 공용 컴포넌트

기존 공용 컴포넌트 경로(`components/common/dynamic-form-builder.tsx`·`dynamic-form-renderer.tsx`·`form-schema.ts`)는 그대로 유지하되 내부 구현을 신규 그리드 엔진으로 전면 재작성한다(form.io 관련 타입·`FORM_BUILDER_OPTIONS` 폐기).

| 컴포넌트 | 역할 |
|---|---|
| `dynamic-form-builder.tsx` | "Form 설정" 팝업(좌 팔레트/우 8×n 그리드 캔버스)을 감싼 관리자용 폼 설계 컴포넌트. **캔버스=미리보기 통합** — 각 캔버스 카드가 그 컴포넌트의 실제 렌더링 모습을 그대로 보여주며(비활성·순수 시각적, `GridComponentBody` 재사용), 별도 미리보기 패널은 없다. 팔레트 항목은 클릭 배치와 드래그앤드롭 배치를 모두 지원한다(5.2절, 2026-07-18 후속 유지보수 요청 4차). 팔레트 상단에는 "라벨 추가" 버튼과 생성된 라벨 칩 목록이 있다(5.8절). `initialSchema`(기존 저장된 그리드 스키마)로 편집 모드 진입, 팝업 "적용" 클릭 시 최신 그리드 스키마를 상위 상태에 반영(자동저장 없음) |
| `dynamic-form-renderer.tsx` | 그리드 스키마를 그대로 렌더링하는 요청자용 컴포넌트. 제출 시 컴포넌트 `key` 기준 값 맵을 상위로 전달(클라이언트 검증·오류 표시는 5.5절, 서버 재검증은 [api_spec/common.md](../api_spec/common.md) 0-2절) |
| `form-schema.ts` | 신규 그리드 스키마 타입(컴포넌트 배열, 위치/크기/Content 설정, `guide-text`/`guide-file` 정적 타입 포함, 컴포넌트별 선택적 `labelId` 참조, 최상위 `labels` 배열 — 5.8절)과 5.3절 팔레트 상수만 정의. 기존 Form.io Form JSON 타입 별칭은 폐기. 그리드 직접 배치 타입이던 `label`은 폐기되고 5.8절의 라벨(태그)로 대체 |

### 5.2 그리드 규칙

- 캔버스는 가로 8칸 고정, 세로는 고정 행 수 없이 스크롤 가능한 동적 증가(컴포넌트를 배치할 때마다 필요한 만큼 아래로 확장).
- 모든 컴포넌트는 폭 1~2칸으로 제한한다. 높이 상한은 유형별로 다르다(2026-07-18 후속 유지보수 요청 4차로 세분화): `text`/`date`/`file`/`select`/`guide-file`는 **1칸 고정**(2칸 불가), `radio`/`checkbox`/`guide-text`는 기존대로 1~2칸, `textarea`는 높이 제약이 없다(폭은 모두 동일하게 1~2칸). 이 변경 전에 이미 2칸 높이로 저장된 `text`/`date`/`file`/`select`/`guide-file` 컴포넌트는 자동으로 축소하지 않는다(로드 시 강제 클램프 없음) — 관리자가 빌더에서 다시 열어 리사이즈해야 새 상한이 적용된다(재량 결정, 저장된 레이아웃을 예고 없이 바꾸지 않기 위함).
- 배치·리사이즈는 1칸 단위로 스냅한다(리사이즈는 컴포넌트 모서리 드래그).
- **배치 방식(2026-07-18 후속 유지보수 요청 4차)**: 팔레트 항목 클릭(기존, 첫 번째 빈 칸에 자동 배치)과 팔레트 항목을 캔버스로 드래그해 원하는 위치에 직접 놓는 드래그앤드롭 배치를 모두 지원한다(둘 다 유지, 클릭은 빠른 추가용·드래그는 정확한 위치 지정용). 드래그 중에는 캔버스 이동/리사이즈와 동일한 1칸 스냅 미리보기를 보여주고, 드롭 위치가 다른 컴포넌트와 겹치면 배치를 막고 기존과 동일한 겹침 경고를 표시한다(캔버스 밖에 놓으면 배치 취소).
- 이미 다른 컴포넌트가 점유한 셀에 겹쳐 배치를 시도하면 배치를 막고 인라인 안내("이미 배치된 컴포넌트와 겹칩니다")를 표시한다.
- 그리드 셀 가로 폭은 팝업 캔버스 폭을 8등분한 반응형 값, 세로 행 높이는 고정값을 사용한다(구체 px 값은 FE 구현 시 결정).
- **빌더 캔버스 카드 = 실제 렌더링(캔버스=미리보기 통합, 2026-07-18 후속 유지보수 요청 정정)**: 캔버스에 배치된 카드는 빈 점선 placeholder나 컴포넌트 유형 아이콘/캡션이 아니라, 그 컴포넌트가 요청 폼에서 실제로 렌더링될 모습을 카드 콘텐츠 영역에 그대로 보여준다(`GridComponentBody` 재사용 — Content 설정 팝업의 개별 미리보기·요청 제출 폼 렌더러와 동일 로직). 이 렌더링은 **순수 시각적 표현**이며 값 입력·상호작용은 되지 않는다(카드 자체가 여전히 드래그 이동 대상이므로 렌더링된 컨트롤과의 상호작용은 의도적으로 막는다). 실제 값 입력을 테스트하려면 기존 Content 설정 팝업 안 개별 실시간 미리보기(5.4절, 상호작용 가능)를 사용한다. 카드 헤더의 **컴포넌트 유형 식별 아이콘(타입별 아이콘)은 제거**한다 — 실제 렌더링 내용이 그 역할을 대체하므로 아이콘·유형명 캡션 텍스트 없이 콘텐츠만 표시한다(기존 "1×1 크기면 아이콘만 표시" 규칙은 유형 아이콘 자체가 없어져 자연히 폐기됨). 1×1처럼 좁은 셀에서의 렌더링 축소 표시는 `GridComponentBody`가 이미 갖고 있는 오버플로우 처리(옵션 자동 줄바꿈+셀 내부 스크롤, 5.4절 렌더링 요건)를 그대로 재사용하며 별도 축소 로직을 두지 않는다. 카드 hover 시 노출되는 설정(`Settings2`)·삭제(`Trash2`) 액션 버튼은 렌더링된 콘텐츠 위에 오버레이 형태로 띄운다(콘텐츠가 카드 전체를 채우므로 별도 헤더 바를 두지 않고 공간을 아낌, 액션 버튼 클릭은 기존과 동일하게 드래그 시작과 분리 처리). 컴포넌트에 라벨(태그)이 지정된 경우의 경계 테두리는 카드 자체가 아니라 캔버스 위 별도 오버레이 레이어로 그린다(카드 콘텐츠 렌더링과는 독립적인 레이어 — 5.8절).

### 5.3 팔레트 컴포넌트 (9종)

`text`(기존 number/email/phone 등 세부 유형은 별도로 두지 않고 text로 통합, 형식 제약은 5.4절 정규식으로 커버) / `textarea` / `select` / `radio` / `checkbox` / `date` / `file` / `guide-text`(정적 안내 텍스트, 값 입력 없음 — 5.4절) / `guide-file`(정적 첨부 가이드 파일, 값 입력 없음 — 5.4절).

> **2026-07-18 후속 유지보수 요청**: 직전 유지보수(20260718-175042)에서 신규 도입한 `guide`(안내 텍스트+첨부 파일 통합) 타입을 폐기하고, 안내 텍스트 전용 `guide-text`와 첨부 파일 전용 `guide-file` 2종으로 분리한다. `guide` 타입으로 저장된 운영 데이터는 없음(테스트 픽스처만 존재)이 확인되어 별도 마이그레이션 없이 코드에서 완전히 제거한다(5.7절).

> **2026-07-18 후속 유지보수 요청 4차**: 그리드에 직접 배치하던 정적 컴포넌트 `label`을 팔레트에서 제거한다. "라벨"은 더 이상 캔버스에 배치하는 컴포넌트가 아니라, 다른 컴포넌트에 선택적으로 부여하는 태그로 전면 개편됐다(5.8절 — 팔레트 자체 개수는 9종으로 줄었지만 라벨 관리 UI는 팔레트 영역 상단에 별도로 존재).

### 5.4 Content 설정 (컴포넌트별 미니 팝업)

- **기본값**: input 폭은 셀 폭의 90%로 가로 가운데·세로 상단 정렬(9방향 중 "상단-가운데", 아래 정렬 항목 참조).
- **팝업 크기**: 안내/가이드 컴포넌트(텍스트+파일)와 컴포넌트별 개별 미리보기 영역을 함께 수용해야 하므로, 팝업 폭을 기존(320px)보다 확장한다(최소 400px 이상 권장, 구체 px 값은 FE 구현 시 결정). 높이는 기존과 동일하게 가용 높이 내 스크롤(overflow-y) 처리를 유지한다.
- 배치된 컴포넌트에 마우스 hover 시 상단에 설정 버튼이 노출되며, 클릭 시 미니 팝업에서 아래 항목을 조정한다(`guide-text`/`guide-file` 제외 7종 공통): input(폭 %, 정렬 위치, placeholder, default 값, 읽기전용 여부, 필수 여부), (text 전용) Validation 정규식(regex, 선택 입력 — 미지정 시 형식 검증 없음), 라벨(태그) 지정(선택, 5.8절). 필수 여부와 정규식은 별개 항목으로, 필수 체크와 정규식 검증이 함께 지정되면 서버는 둘 다 적용한다.
- **정규식 검증은 `text` 타입 전용(2026-07-18 후속 유지보수 요청 4차로 축소)**: 기존에는 7개 입력 타입 전체에 무관하게 정규식 UI·서버 검증이 적용됐으나, 이제 `type=text`에만 적용한다. Content 설정 팝업의 정규식 입력 UI는 `text` 타입일 때만 노출하고 나머지 6종(textarea/select/radio/checkbox/date/file)에서는 숨긴다. 이 변경 전 다른 타입 컴포넌트에 이미 저장된 `validation.regex` 값은 데이터를 지우거나 마이그레이션하지 않는다(재량 결정) — 스키마에는 남아 있을 수 있으나 UI에 노출되지 않고 서버도 `type=text`가 아니면 더 이상 평가하지 않아 사실상 무시된다.
- **placeholder**: 7종 입력 컴포넌트 모두 스키마상 `input.placeholder` 필드를 가지되(값 미지정 시 null), Content 설정 팝업의 placeholder 입력 UI는 **단일 값 표시 컨트롤을 갖는 유형에만 노출**한다 — `text`/`textarea`/`select`/`date`/`file`. `radio`/`checkbox`는 다중 옵션 목록 UI라 placeholder가 시각적으로 의미가 없으므로 팝업에 노출하지 않는다(스키마 필드 자체는 존재하되 UI 미노출·렌더링 미반영, `options`가 `hasGridOptions` 유형에만 노출되는 것과 동일한 패턴). 렌더링 시 값이 비어 있으면 placeholder를, 값이 있으면 실제 값을 표시한다. 미지정 시 유형별 폴백 문구(예: select는 기존과 동일한 "선택")를 사용한다.
- `select`/`radio`/`checkbox`는 옵션을 콤마(,) 구분 텍스트로 입력해 설정한다. 옵션 설정 UI에는 "CI 연계" 라디오 버튼 자리를 마련해 두되(향후 확장용), 클릭 시 CI 하위 아이템 자동 추가 등 실제 동작은 구현하지 않는다(배치만 확정 — 이번 범위 밖).
- **`radio`/`checkbox` 배치 방향·옵션 간 여백(2026-07-18 후속 유지보수 요청 신규)**: `select`는 대상이 아니며 `radio`/`checkbox`에만 적용한다. Content 설정에 옵션 배치 방향(가로/세로)과 옵션 간 여백 3단계(좁게/보통/넓게)를 추가한다. **기본값은 가로 배치·여백 1단계(좁게)**다(기존에는 세로 배치로 고정되어 있었으므로, 이 기본값 변경으로 기존에 저장된 스키마 중 방향을 명시하지 않은 컴포넌트는 렌더링 시 세로 → 가로로 시각적 동작이 바뀐다 — 별도 데이터 마이그레이션은 없음, 미지정 시 폴백 기본값 적용). 가로 배치 시 옵션이 셀 폭을 넘으면 자동 줄바꿈(wrap)하고, 줄바꿈으로 셀 높이가 늘어나지 않도록 기존 렌더링 요건(아래)과 동일하게 셀 내부 스크롤로 처리한다. 여백 3단계의 실제 px/rem 값은 FE 구현 시 결정한다.
- **`date`/`file` 렌더링(2026-07-18 후속 유지보수 요청 — 직전 아이콘 전용 표시 롤백)**: 아이콘만 남기는 표시를 되돌리고, 다른 입력 컴포넌트와 동일하게 **입력 박스(input.widthPercent/align/verticalAlign 적용)+박스 우측 끝 아이콘**(달력/파일) 형태로 표시한다. 값이 없으면 박스 안에 placeholder 문구를(플레이스홀더 미지정 시 유형별 기본 안내), 값이 있으면 박스 안에 날짜(`yyyy-MM-dd`)/파일명 텍스트를 표시한다. **클릭 동작**: 박스 전체(텍스트 영역+우측 아이콘)가 클릭 영역이며, 어디를 클릭해도 네이티브 date picker/파일 선택 다이얼로그가 열린다(숨겨진 네이티브 `<input type="date"/"file">`을 트리거, 박스 자체는 실제 네이티브 input이 아니라 버튼 역할 — 자유 텍스트 타이핑은 지원하지 않는다). 셀 크기 제약은 폭 1~2칸·높이 1칸 고정(2026-07-18 후속 유지보수 요청 4차, 5.2절)이며, 박스 안 텍스트가 셀 폭 안에서 잘리면 다른 유형과 동일하게 `truncate` 처리한다.
- **렌더링 요건**: `select`/`radio`/`checkbox`의 옵션 텍스트가 길거나 개수가 많을 때 셀 폭(1~2칸) 안에서 잘리지 않도록 옵션을 자동 줄바꿈하고, 셀 높이가 옵션 줄바꿈으로 늘어나지 않도록 셀 내부 스크롤(overflow)로 처리한다(그리드 레이아웃 무결성 유지, 구체 스크롤/축소 방식은 FE 구현 시 결정).
- **정렬 9방향(가로 3×세로 3, 2026-07-18 후속 유지보수 요청 4차로 확장)**: 기존에는 가로(좌/가운데/우)만 지원했으나, 입력 컴포넌트의 `input.align`(셀 안 폭%박스의 위치)과 `guide-text`의 `textAlign`(텍스트 정렬) 둘 다 세로축(상단/가운데/하단)을 추가해 좌상단~우하단 총 9개 위치 중 하나를 선택하도록 확장한다. Content 설정 UI는 기존 가로 3버튼 토글 대신 **3×3 그리드 선택 위젯**(앵커 포인트 선택 UI, 한 번의 클릭으로 가로·세로를 동시에 지정) 하나로 통일한다. 세로 정렬의 기본값은 **상단**(기존 시각적 동작을 그대로 보존하기 위한 재량 결정 — 세로 정렬 개념이 없던 기존 저장 데이터도 상단 정렬로 동일하게 렌더링됨), 가로 정렬의 기본값은 기존과 동일(입력은 가운데, `guide-text`는 좌측). 셀을 꽉 채우는 콘텐츠(예: 옵션이 많아 스크롤되는 `radio`/`checkbox`)는 세로 정렬을 지정해도 시각적으로 차이가 없을 수 있다(정상 동작 — 5.4절 렌더링 요건의 셀 내부 스크롤이 우선).
- **`guide-text`/`guide-file` 컴포넌트(안내/가이드, 값 입력이 없는 정적 컴포넌트)**: 그리드 직접 배치 타입이던 `label`은 폐기되고 라벨(태그, 5.8절)로 대체됐다 — `guide-text`/`guide-file`은 이번 개편 대상이 아니며 계속 그리드에 직접 배치하는 정적 컴포넌트로 유지한다.
  - `guide-text`: Content 설정은 **안내 텍스트**(여러 줄 입력)와 정렬(9방향, 위 항목 참조)만 제공한다. 렌더링 시 안내문임을 시각적으로 구분(예: 안내 아이콘 동반 등, 구체 스타일은 FE 재량)한다. 높이 제약은 1~2칸(`radio`/`checkbox`와 동일, 5.2절).
  - `guide-file`: Content 설정은 **가이드 파일 첨부**(파일 선택, 선택 입력)만 제공하며 텍스트 입력은 없다. 첨부 파일은 카탈로그 항목의 `form_schema`(JSONB)에 **base64 데이터 URL로 인라인 저장**한다(기존 제출 파일(`file` 컴포넌트) 처리 컨벤션과 동일 방식 — 별도 업로드 API·오브젝트 스토리지 신규 도입 없음, [database/service-request.md](../database/service-request.md) 참조). 렌더링(SCR-SRM-002, 5.6절 미리보기)에서는 첨부 파일이 있으면 다운로드 버튼/링크를 노출해 요청자가 파일을 내려받을 수 있게 한다(미첨부 시 빈 상태 안내). 안내 텍스트가 함께 필요하면 관리자가 `guide-text` 컴포넌트를 인접 셀에 별도 배치한다(둘 사이 접근성 연결 없음). 높이 제약은 1칸 고정(2026-07-18 후속 유지보수 요청 4차, 5.2절).
  - 둘 다 제출 데이터에는 포함되지 않으며(조회 전용), 유효성 검증·제출 데이터 대상에서 제외한다.
- **라벨(태그) 지정(2026-07-18 후속 유지보수 요청 4차 신규)**: 7종 입력 컴포넌트와 `guide-text`/`guide-file` 모두 Content 설정에서 이미 생성된 라벨(5.8절) 중 하나를 지정하거나 "없음"으로 해제할 수 있다(Select, 후보는 5.8절에서 생성한 라벨 목록). 라벨 생성 자체는 이 팝업이 아니라 팔레트 상단 "라벨 추가" 버튼에서 한다(5.8절).
- **개별 컴포넌트 실시간 미리보기(2026-07-18 후속 유지보수 요청 신규)**: Content 설정 팝업 내부에 편집 중인 컴포넌트 **단독**의 렌더링 미리보기 영역을 추가한다(5.2절의 캔버스 카드 렌더링과는 별개 기능 — 캔버스 카드 렌더링은 순수 시각적 표현으로 상호작용이 안 되지만, 이 Content 설정 팝업 안 미리보기는 **상호작용 가능**해 실제 값 입력을 테스트할 수 있다). placeholder·정렬(9방향)·기본값·읽기전용·옵션·(text 전용) 정규식·(radio/checkbox) 배치 방향·여백 등 팝업에서 설정을 변경하면 미리보기가 즉시(입력 즉시) 갱신된다. 미리보기 안에서 값을 입력해볼 수 있으나 이는 미리보기 전용 로컬 상태이며 `defaultValue`나 실제 제출 데이터에는 반영되지 않는다(순수 확인용). FE는 그리드 렌더러(`dynamic-form-renderer.tsx`)가 이미 갖고 있는 유형별 컨트롤 렌더링 로직을 그대로 재사용해 빌더 팝업에 노출한다(로직 중복 구현 금지). 이 미리보기에는 라벨 경계 테두리(5.8절, 캔버스 전용 오버레이)를 표시하지 않는다.

### 5.5 유효성 검증·오류 표시

- 실시간(입력 중) 검증은 하지 않는다. 제출 버튼을 클릭할 때마다 `formSchema.components` 배열 순서대로(그리드 시각적 위치가 아니라 스키마 배열 삽입 순서 기준) 각 컴포넌트를 검사한다.
- 배열 순서상 **첫 번째로 검증에 실패하는 컴포넌트의 오류 메시지 1건만** 폼 하단(제출/취소 버튼 위 공통 위치)에 표시하고 제출을 막는다. 필드별 인라인 오류 표시는 없다.
- 사용자가 해당 필드를 고친 뒤 제출 버튼을 다시 클릭하면, 그 시점에 배열 처음부터 다시 순서대로 검사해 다음으로 실패하는 컴포넌트의 오류 1건을 표시한다(한 번의 클릭마다 한 단계씩 진행, 여러 오류를 한 번에 모아 보여주지 않음).
- 모든 컴포넌트가 통과해야 제출 데이터가 상위로 전달된다.
- **서버 재검증([api_spec/common.md](../api_spec/common.md) 0-2절)도 동일한 계약을 따른다** — `components` 배열 순서대로 검사해 첫 번째 위반 컴포넌트에서 즉시 400을 반환한다(여러 위반을 모아 반환하지 않음, 클라이언트와 동일한 "순차 1건" 원칙 유지).

### 5.6 팝업 레이아웃(캔버스=미리보기 통합)·저장 흐름

- **캔버스=미리보기 통합(2026-07-18 후속 유지보수 요청 정정)**: "Form 설정" 팝업(모달) 내부는 **좌 팔레트 / 우 8×n 그리드 캔버스** 2분할이다. 직전에 도입했던 별도 우측 "전체 레이아웃 미리보기" 패널은 사용자 피드백으로 폐기한다 — 미리보기를 별도 패널로 분리하는 대신, **캔버스 자체가 곧 미리보기**가 되도록 각 카드가 실제 렌더링 모습을 직접 보여준다(5.2절). 기존에 SCR-SRM-007 카탈로그 항목 편집 폼(팝업 바깥)에 있던 45% 축소 pre-view 축소판도 이번 개편으로 완전히 제거되며 별도 대체물을 두지 않는다(캔버스 자체가 실시간 렌더링을 보여주므로 축소판이 불필요).
  - 캔버스 카드의 렌더링은 편집 중(적용 전)인 `components` 상태를 실시간 반영한다 — 배치·리사이즈·Content 설정 변경이 즉시 해당 카드에 나타난다.
  - `guide-text`/`guide-file` 렌더링 규칙, `date`/`file` 입력 박스+우측 아이콘 렌더링 규칙, radio/checkbox 배치 방향·여백 설정, 정렬 9방향(5.4절)도 캔버스 카드 렌더링에 동일하게 적용한다. 라벨(태그) 경계 테두리(5.8절)는 카드가 아니라 캔버스 전체 위 오버레이 레이어로 그린다.
  - **팝업(모달) 크기**: 더 이상 별도 미리보기 패널을 수용할 필요가 없으므로, 3분할 대비 폭 확대가 강제되지는 않는다(팔레트+캔버스 2분할 기준으로 적절한 크기를 FE가 결정 — 기존 2분할 크기로 되돌리거나, 카드 안 실제 렌더링 콘텐츠(옵션 목록 등)가 편하게 보이도록 유지·조정하는 것 모두 FE 재량). 컴포넌트별 Content 설정 미니 팝업(420px, 5.4절)과는 별개의 팝업(모달)이다.
  - **라벨 칩 영역 위치(2026-07-18 후속 유지보수 요청 4차 신규)**: 팝업 모달 타이틀("Form 설정")과 팔레트+캔버스 2분할 영역 사이에 생성된 라벨(태그) 칩 목록을 가로로 표시하는 별도 스트립을 둔다(5.8절). 라벨 생성 버튼 자체는 팔레트 컬럼 최상단에 있다(5.8절).
- 팝업 우측 하단 적용/취소 버튼. 적용 시 팝업이 닫히고 그리드 스키마가 상위 상태(카탈로그 항목 편집 폼)에 반영된다 — 실제 API 저장은 기존과 동일하게 카탈로그 항목의 명시적 "저장" 버튼이 담당한다.
- 재편집(팝업 재오픈) 시 직전 저장된 스키마 그대로 캔버스에 재렌더링한다.

### 5.7 기존 데이터 처리

기존 form.io Form JSON(`service_catalog_item.form_schema`)은 신규 그리드 스키마와 구조가 호환되지 않아 자동 마이그레이션이 불가능하므로, 배포 시 전체 로우를 신규 스키마의 초기값(빈 그리드)으로 리셋한다(사용자 승인 완료 — [database/service-request.md](../database/service-request.md) 참조). 프로세스 오너는 배포 후 각 카탈로그 항목의 폼을 새 빌더로 다시 구성해야 한다.

**`guide` 타입 폐기(2026-07-18 후속 유지보수 요청)**: 직전 유지보수(20260718-175042)에서 도입한 `guide` 타입은 코드에서 완전히 제거하고 `guide-text`/`guide-file`로 대체한다. DB/BE 확인 결과 `guide` 타입으로 저장된 운영 데이터가 없어(테스트 픽스처만 존재) 별도 백필·마이그레이션 스크립트는 필요 없다.

**그리드 배치형 `label` 타입 폐기(2026-07-18 후속 유지보수 요청 4차)**: 5.8절의 라벨(태그) 개편으로 그리드에 직접 배치하던 `label` 타입은 완전히 제거된다. `type: "label"` 컴포넌트가 하나라도 포함된 `service_catalog_item.form_schema` 로우는 배포 시 초기값(빈 그리드, `{"components":[],"labels":[]}`)으로 리셋한다(사용자 확답 완료 — 통합테스트로 생성된 데이터 외 운영 데이터 없음). `label`을 사용하지 않은 로우(다른 9종 컴포넌트만 사용)는 이 리셋 대상이 아니다.

### 5.8 라벨(태그) 관리

기존에 그리드에 직접 배치하던 정적 `label` 컴포넌트를 폐기하고, **다른 컴포넌트에 선택적으로 부여하는 태그**로 전면 개편한다(2026-07-18 후속 유지보수 요청 4차). `guide-text`는 이 개편 대상이 아니며 계속 그리드 직접 배치 정적 컴포넌트로 유지한다(5.3/5.4절).

**데이터 모델**: `GridFormSchema`에 최상위 배열 `labels`(각 항목은 `id`/`text`(라벨 텍스트)/`textColor`(글자색)/`borderColor`(테두리색))를 신설한다. 7종 입력 컴포넌트와 `guide-text`/`guide-file`은 선택적으로 `labelId`(하나의 라벨 참조, 미지정 시 null)를 가진다. 라벨은 컴포넌트와 별개인 독립 엔티티이며, 컴포넌트 삭제 시 그 컴포넌트의 `labelId` 참조만 사라질 뿐 라벨 자체(및 그 라벨을 참조하는 다른 컴포넌트)는 유지된다.

**생성 UI**: 팔레트 컬럼 최상단에 "라벨 추가" 버튼을 둔다(팔레트의 9종 컴포넌트 항목보다 위, 5.1절). 클릭 시 미니 팝업에서 라벨 텍스트, 글자색(textColor), 테두리색(borderColor)을 입력한다(색상은 색상 코드 입력 또는 `<input type="color">` — 구체 UI는 FE 재량). 이 팝업은 라벨 자체의 속성만 관리하며, 컴포넌트에 지정하는 것은 5.4절의 Content 설정 팝업에서 별도로 한다(B9).

**표시·관리 UI**: 생성된 라벨은 팝업 모달 타이틀과 팔레트+캔버스 2분할 영역 사이의 가로 스트립에 칩(rounded-square, 배경은 옅은 tint, 텍스트는 `textColor`, 테두리는 `borderColor`) 형태로 나열한다(5.6절). 라벨이 많아지면 가로 스크롤로 처리한다(구체 방식 FE 재량). 칩 클릭 시 생성 팝업과 동일한 미니 팝업이 값 프리필로 재오픈되어 텍스트·색상을 수정할 수 있고, 칩에 삭제(×) 버튼을 둬 라벨 자체를 삭제할 수 있다(**설계 추가 사항** — 이 시스템의 다른 목록형 관리 UI(예: SCR-SRM-009 카테고리 관리)와 동일하게 생성만 있고 수정·삭제가 없는 반쪽짜리 구현을 피하기 위해 designer가 보강). 라벨 삭제 시 그 라벨을 참조하던 모든 컴포넌트의 `labelId`는 null로 해제된다(컴포넌트 자체는 삭제되지 않음).

**컴포넌트에 라벨 지정(B9)**: Content 설정 팝업(5.4절)에서 이미 생성된 라벨 목록 중 하나를 Select로 선택하거나 "없음"으로 해제한다. 하나의 컴포넌트는 라벨을 최대 1개까지만 가질 수 있다(다중 라벨 지정 없음).

**캔버스 경계 테두리 표시(B5)**: 캔버스는 라벨별로 그 라벨이 지정된 컴포넌트들의 위치를 계산해 경계 테두리를 오버레이로 그린다(카드 자체가 아니라 캔버스 위 별도 레이어, 5.2/5.6절 — 그리드 좌표→픽셀 변환은 기존 셀 크기 계산 로직을 재사용).
- 라벨이 **컴포넌트 0개 또는 1개**에만 지정되면 테두리를 그리지 않는다(1개뿐이면 "경계"의 의미가 없으므로).
- 라벨이 **컴포넌트 2개 이상**에 지정되면, 그 컴포넌트들의 그리드 좌표에서 최소 col/row(topX/topY)와 최대 col+w/row+h(bottomX/bottomY)를 계산해 그 사각형 전체를 테두리로 그린다(**확정 사항**: 이 사각형은 라벨이 지정되지 않은 다른 셀·다른 컴포넌트가 사각형 내부에 포함되더라도 그대로 시각적으로 테두리 안에 포함되는 것을 허용한다 — 사각형을 좁히거나 라벨 미지정 셀을 제외하는 별도 로직 없음).
- 테두리 색상은 그 라벨의 `borderColor`를 사용한다.
- 컴포넌트 이동·리사이즈·라벨 지정/해제 시 해당 라벨의 경계 사각형을 즉시 재계산한다(캔버스 카드 렌더링과 동일하게 편집 중 상태를 실시간 반영).
- 이 테두리 오버레이는 캔버스에만 표시되며, Content 설정 팝업 안 개별 미리보기(5.4절)나 SCR-SRM-002 요청 제출 폼·SCR-SRM-007 화면 자체의 축소 미리보기(A1, 4절)에는 표시하지 않는다(관리자의 편집 보조 용도로 한정, 요청자에게는 노출되지 않는 순수 편집 UI 요소).

**기존 데이터 처리**: 5.7절 참조 — `type: "label"` 컴포넌트를 포함한 기존 `form_schema` 로우는 리셋 대상이다.
