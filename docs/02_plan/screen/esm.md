## 1. 개요

부서별(HR/법무/시설/재무) 서비스 카탈로그·요청 처리, HR 케이스 관리(HR 전용), 온보딩/오프보딩 체크리스트(부서 간 하위 작업 오케스트레이션·진행 추적), ESM 지표 대시보드 화면을 정의한다. 요청 제출·처리 흐름은 서비스 요청 관리(`docs/02_plan/screen/service-request.md`)의 UX 패턴을 재사용하되 담당 부서로 구분한다.

## 4. 화면 상세

### SCR-ESM-001 · 부서 서비스 포털(카탈로그 브라우즈)

- **목적**: 요청자가 부서별 서비스 카탈로그에서 요청 유형을 탐색·선택.
- **레이아웃**: 상단 부서 탭(HR/법무/시설/재무) + 검색바 / 카테고리 카드 그리드.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 부서 탭 | 탭 | 담당 부서별 카탈로그 필터 | Base(활성) |
  | 카탈로그 카드 | 카드 | 요청 유형 선택 진입 | Base(강조) |
  | 검색바 | 입력 | 유형/키워드 검색 | Border/Base |
- **상태 · 인터랙션**: 카드 클릭 시 SCR-ESM-002 이동.
- **연관 API**: `GET /api/v1/esm/catalog-items?department=`

### SCR-ESM-002 · 부서 요청 제출(동적 양식)

- **목적**: 선택한 요청 유형의 동적 양식을 작성해 제출.
- **레이아웃**: 동적 폼(카탈로그 항목의 `form_schema` 렌더 — 컬럼/패널/탭 등 관리자가 설계한 레이아웃 그대로) / 하단 제출 버튼.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 동적 폼 렌더러 | `@formio/react` `Form`([common.md](common.md) 8절 공용 `dynamic-form-renderer.tsx`, SRM SCR-SRM-002와 동일 컴포넌트 재사용) | `formSchema`(API-ESM-002) 그대로 렌더, 클라이언트 검증 내장 | Border/Base |
  | 제출 버튼 | 버튼 | `Form`의 `onSubmit`으로 제출 데이터 수신 후 요청 생성 호출 | Base |
- **상태 · 인터랙션**: 필수·형식 위반 필드 미입력 시 제출 차단·인라인 오류(Form.io 내장 검증). 온보딩/오프보딩 유형 제출 성공 시 체크리스트 자동 생성 안내 토스트 노출. 서버 재검증 실패(400) 시 공통 오류 토스트로 안내.
- **연관 API**: `GET /api/v1/esm/catalog-items/{id}`, `POST /api/v1/esm/requests`

- **목적**: 요청자가 본인이 제출한 부서 요청 상태를 추적.
- **레이아웃**: [SCR-COM-007] 목록 패턴. 필터(부서·상태·기간) / 표(접수번호·부서·유형·상태·갱신일).
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 부서 배지 | Lozenge | 담당 부서 표시 | Info |
  | 상태 배지 | Lozenge | 제출/처리중/완료/반려 | 상태 시맨틱 |
- **상태 · 인터랙션**: 본인 요청만 조회. 행 클릭 시 상세.
- **컬럼 폭([common.md](common.md) SCR-COM-007 아키텍처 적용)**:
  | 컬럼 | 폭(px) | 근거 |
  |------|--------|------|
  | 접수번호 | 130 | 코드(ticketKey) |
  | 부서 | 100 | 부서 뱃지(짧음) |

  | 상태 | 110 | 상태 뱃지 |
  | 갱신일 | 110 | 날짜(YYYY-MM-DD) |
- **페이지당 아이템 수**: 13건([common.md](common.md) 7절 패턴 A).
- **연관 API**: `GET /api/v1/esm/requests?scope=mine`

- **컬럼 폭([common.md](common.md) SCR-COM-007 아키텍처 적용)**:

  | 갱신일 | 110 | 날짜(YYYY-MM-DD) |

### SCR-ESM-005 · 부서 요청 상세

- **목적**: 부서 요청 처리(상태 전이·코멘트) 및 연계된 온보딩/오프보딩 체크리스트 확인.
- **레이아웃**: [SCR-COM-008] 상세 패턴. 본문(양식 값)·코멘트·타임라인 / 우측 메타(상태·담당 부서·연계 체크리스트 요약).
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 상태 전이 버튼 | 버튼 | 처리중/완료/반려. 라벨은 동작 동사형([common.md](common.md) SCR-COM-008 아키텍처 적용) — 아래 표 | Base/Success/Danger |
  | 연계 체크리스트 카드 | 카드 | 온보딩/오프보딩 시 진행률 요약, 클릭 시 SCR-ESM-009 이동 | Info |
  | 코멘트 입력 | 입력 | 요청자-처리자 소통 | Border/Base |
  | 승인 패널(공용) | 패널 | COMPLETED 전이에 매칭되는 승인 프로세스가 있으면 차수 진행 상태(API-COM-004) 표시, 없으면 패널 자체를 노출하지 않고 기존처럼 즉시 전이 | Info/Warning/Success |
- **상태 · 인터랙션**: 담당 부서 처리자가 아니면 상태 전이 버튼 비노출. 체크리스트가 없는 일반 요청은 카드 미노출. COMPLETED 전이 시도 시 매칭되는 승인 프로세스가 있으면 409와 함께 승인 패널이 나타나며, 처리는 [common.md](common.md) SCR-COM-014에서 수행한다.
  - **전이 버튼 라벨**: `status.ts`의 `transitionLabel(t, target)`(i18n 키 `esm:transition.*`)이 버튼 텍스트를 결정한다(`requestStatusLabel(t, target)`은 배지·토스트 문구에 별도로 사용).
    | 도착 상태 | 버튼 라벨 |
    |-----------|-----------|
    | IN_PROGRESS | 처리 시작 |
    | COMPLETED | 완료 처리 |
    | REJECTED | 반려 처리 |
  - **타임라인 actor 표시**: `RequestDetailResponse.TimelineEntry`(esm 패키지)의 `actor` 필드가 행위 수행 주체자를 표시한다([common.md](common.md) SCR-COM-008 아키텍처 참조). 상태 변경 타임라인 메시지는 `target.name()`(코드) 대신 상태 라벨을 사용한다.
- **연관 API**: `GET /api/v1/esm/requests/{id}`, `PATCH .../status`, `POST .../comments`, `GET /api/v1/approvals/{approvalRequestId}`(API-COM-004)

### SCR-ESM-006 · 부서별 카탈로그 관리

- **목적**: 프로세스 오너가 담당 부서 지정 요청 유형(양식·체크리스트 템플릿)을 정의·정리.
- **레이아웃**: 카탈로그 항목 목록 / 항목 편집 폼(담당 부서·이름·설명·양식 필드 빌더·체크리스트 템플릿 빌더).
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 담당 부서 선택 | 셀렉트 | HR/법무/시설/재무 | Border |
  | 양식 폼 빌더 | `@formio/react` `FormBuilder`([common.md](common.md) 8절 공용 `dynamic-form-builder.tsx`, SRM SCR-SRM-007과 동일 컴포넌트 재사용) | 드래그앤드롭 자유배치(컬럼/패널/탭 레이아웃 포함) 폼 설계. 팔레트는 8.2절 구성. 기존 조회 스키마로 편집 모드 진입, `onChange` 축적 후 "폼 저장" 버튼으로 반영 | Border/Base |
  | 체크리스트 템플릿 빌더 | 반복 입력 | 온보딩/오프보딩 유형 선택 시 노출, 하위 작업(담당 부서·설명) 정의 | Border/Base |
- **상태 · 인터랙션**: 담당 부서 미지정 시 400. 온보딩/오프보딩 유형인데 체크리스트 템플릿이 비어 있으면 저장은 허용하되 경고 표시(실제 제출 시 REQ-ESM-005 Unwanted 조건으로 거부됨을 안내). 프로세스 오너 권한 없으면 403.
- **연관 API**: `GET/POST/PATCH /api/v1/esm/catalog-items`

### SCR-ESM-007 · HR 케이스 목록

- **목적**: HR 담당자가 HR 케이스를 조회·접수.
- **레이아웃**: [SCR-COM-007] 목록 패턴. 필터(상태) / 표(케이스ID·상태·최종 갱신일). 우측 상단 "케이스 접수" 버튼.
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|
  | 상태 배지 | Lozenge | 접수/기록/조사/해결 | 상태 시맨틱 |
  | 케이스 접수 버튼 | 버튼 | 신규 케이스 생성 | Base |
- **상태 · 인터랙션**: HR 케이스 매니저가 아니면 화면 진입 자체가 403(사이드바 미노출).
- **컬럼 폭([common.md](common.md) SCR-COM-007 아키텍처 적용)**:
  | 컬럼 | 폭(px) | 근거 |
  |------|--------|------|
  | 케이스ID | 130 | 코드(`HR-{id}`) |
  | 제목 | 미지정(auto) | 길이 가변, 잔여 폭 흡수. `truncate` 적용 필요 |
  | 상태 | 110 | 상태 뱃지 |
  | 최종 갱신일 | 110 | 날짜(YYYY-MM-DD) |
- **페이지당 아이템 수**: 13건([common.md](common.md) 7절 패턴 A).
- **연관 API**: `GET /api/v1/esm/hr-cases`, `POST /api/v1/esm/hr-cases`

### SCR-ESM-008 · HR 케이스 상세

- **목적**: HR 케이스를 접수→기록→조사→해결 단계로 처리.
- **레이아웃**: [SCR-COM-008] 상세 패턴(민감정보 특성상 연결 항목 패널 제외). 본문(내용)·상태 이력 / 우측 메타(대상자·상태).
- **구성 요소**:
  | 요소 | 유형 | 설명 | 색상 |
  |------|------|------|------|

### SCR-ESM-009 · 온보딩/오프보딩 체크리스트 상세

  | 전체 진행률 배지 | Lozenge | 진행중/완료 | Warning/Success |
