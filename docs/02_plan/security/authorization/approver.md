# 역할 정의 — APPROVER

> 역할: APPROVER · 버전: 0.4 · 작성일: 2026-07-15 · 개발 중 발견(dev-lead) 반영 — ASSET 상세조회는 PROBLEM/VULNERABILITY/COMPLIANCE/ESM과 달리 원래부터 역할 제한이 없는 **의도된 설계**(`AssetService` 클래스 주석 확인)이므로 매니저 전용 그룹에서 분리, INCIDENT처럼 "이미 개방 상태 — 동적 판정 no-op"으로 정정(단, INCIDENT와 달리 ASSET은 결함이 아니라 의도된 설계라 백엔드 축소 변경 자체를 하지 않음)
>
> 이전 버전: 승인 대상자 역할 기반 동적 상세조회 권한(유지보수 요청) 반영 — SRM/CHANGE의 정적 "도메인 전체조회" 권한을 폐지하고, 승인 프로세스 규칙(도메인/요청유형/요청자역할 3축 매칭)의 승인자 역할(step role) 보유 여부로 판정하는 동적 규칙으로 완전 대체. INCIDENT/PROBLEM/ASSET/VULNERABILITY/COMPLIANCE/ESM 6개 도메인 상세조회에도 동일 규칙 신규 적용(총 8개 도메인). INCIDENT는 기존 백엔드 역할체크 누락 결함(전체조회 가능 상태)도 이번에 함께 정리(Main 확인)
>
> 이전 버전: 승인 프로세스 커스텀 기능(유지보수 요청) 반영 — SRM/CHANGE 전용 고정 승인자에서 "승인 프로세스 규칙에 지정될 수 있는 범용 역할"로 일반화

## 공통 기본 접근 (전 역할 공통)

앱 셸/공통 컴포넌트(SCR-COM-*), 로그인·프로필·비밀번호(SCR-AUTH-001~003), 404, 자기 인증 API(API-AUTH-003/004/005), 내 메뉴 조회(API-AUTH-022), 통합 검색 API(API-SEARCH-001, 접근 가능 도메인만 결과 포함). SYSTEM_ADMIN은 모든 역할의 화면·API에 예외적으로 접근 가능하다(system_admin.md 2절 참조). 이하 역할 고유 접근만 기술한다.

## 1. 페르소나

- **역할명**: APPROVER (승인자 · CAB 멤버 포함)
- **설명**: SYSTEM_ADMIN이 정의한 승인 프로세스 규칙(SCR-ADMIN-008)의 승인자 차수에 지정될 수 있는 범용 역할. 관리자가 어느 도메인·요청유형·차수에 APPROVER 역할을 지정했는지에 따라 실제로 승인하는 대상과 조회 가능한 티켓이 달라진다.
- **주요 목표**: 본인 역할이 지정된 승인 차수를 검토해 승인/반려하고 사유를 남기며, 그 판단에 필요한 티켓 상세를 조회한다.

## 4. 접근 제어 규칙

- 로그인 필요 화면/요청 접근 시 Access Token의 `role`을 검증한다.
- **SCR-COM-014·API-COM-003/004는 역할 제한 없이 인증된 모든 사용자가 접근할 수 있다**(화면·API 자체에 대한 403은 없음). 대신 **결정 처리(API-COM-005)는 인스턴스별 동적 판정**을 거친다: 대상 승인 인스턴스의 현재 대기 차수가 요구하는 역할 목록에 로그인 사용자의 role claim이 하나도 포함되지 않으면 → **403**. 포함되면 처리 가능(공유 대기함, 먼저 처리한 사용자 또는 AND 집계 기준으로 결정 — [common.md](../../02_plan/database/common.md) `approval_decision` 참조).
- **동적 상세조회(2026-07-15 유지보수 요청, [api_spec/common.md](../../02_plan/api_spec/common.md) 0-1절 판정 로직)**: SRM/CHANGE/INCIDENT/PROBLEM/ASSET/VULNERABILITY/COMPLIANCE/ESM 8개 도메인의 티켓 상세조회는, 대상 티켓의 (도메인, 요청유형, 요청자 보유 역할)에 매칭되는 승인 프로세스 규칙이 존재하고 그 규칙의 승인자 차수(어느 차수든) 중 하나라도 로그인 사용자의 role claim과 겹치는 역할이 지정되어 있으면 조회를 허용한다(그 외 접근 조건 — 요청자 본인·도메인 매니저 — 은 각 도메인 문서 그대로 유지, 이 조건과 OR). 매칭되지 않으면 **403**. 실제 승인 인스턴스 생성 여부·현재 진행 차수와 무관하게 규칙 설정 자체와의 매칭으로 판정한다(인스턴스 생성 전에도 조회 가능).
  - **SRM/CHANGE**: 기존 "APPROVER면 무조건 도메인 전체조회 가능"이던 정적 권한은 폐지되었다 — 매칭되지 않는 서비스요청/변경 건은 APPROVER라도 403이다.
  - **PROBLEM/VULNERABILITY/COMPLIANCE/ESM**: 기존에는 APPROVER의 조회 권한 자체가 없었다(각 도메인 매니저 전용). 이번에 매칭 시에만 조회를 허용하는 권한이 신규로 생겼다.
  - **INCIDENT**: 기존에는 백엔드에 역할 제한이 전혀 없어 인증된 사용자면 누구나(FE 라우트 가드만 SERVICE_DESK_AGENT/INCIDENT_MANAGER로 필터링) API 직접 호출 시 전체조회가 가능한 결함성 상태였다. 이번 유지보수로 다른 도메인과 동일하게 백엔드에 `SERVICE_DESK_AGENT`/`INCIDENT_MANAGER` 명시적 역할 체크를 신설하고, 이 동적 판정을 OR로 추가한다(Main 확인 완료 — 결함 정리를 이번 범위에 포함). 매칭되지 않고 두 역할도 없으면 APPROVER는 403이다.
  - **ASSET**: INCIDENT와 겉보기 상태(백엔드 역할 제한 없음)는 같지만 성격이 다르다 — `AssetService` 클래스 주석에 "조회·CI·연계·지표는 인증된 사용자 전반 허용"이 **의도된 설계**로 명시되어 있다(개발 중 발견, 2026-07-15). 따라서 INCIDENT와 달리 이번 유지보수에서 ASSET_MANAGER 전용으로 좁히는 변경은 포함하지 않는다 — 기존처럼 인증된 사용자 전반 허용을 유지하며, APPROVER는 매칭 여부와 무관하게 이미 조회 가능하다(동적 판정 자체가 no-op, 코드 변경 불필요).
- 반려 시 사유 필수(누락 시 400). 이미 결정된 역할 슬롯 재처리 또는 이미 종료된 인스턴스 처리 시도 → 409.
- 화면 이동마다 Backend에 권한 확인 API를 호출한다.
